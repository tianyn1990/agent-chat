# 设计文档：执行状态摘要侧栏与沉浸式工作台

## Context

真实 `Star-Office-UI` 是完整场景页，而不是局部组件。当前右侧侧栏虽然可以技术上嵌入 iframe，但体验存在天然上限：

- 上游页面按固定舞台尺寸设计
- 聊天主界面和右侧侧栏会与真实办公室页面抢宽度
- 缩放后点击、阅读和滚动体验都变差

此外，真实 `Star-Office-UI` 初始化成本不低：

- 需要加载图片、字体、脚本和 Phaser 运行时
- 首次渲染时还会进行资源初始化和状态轮询

如果每次关闭都销毁、再次打开都重建，会造成明显的重复等待。

## Goals / Non-Goals

- Goals:
  - 默认不再在侧栏中承载完整真实像素办公室
  - 提供独立沉浸式工作台来展示真实 iframe
  - 工作台关闭时保留已初始化的 iframe，二次打开直接复用
  - 摘要侧栏与沉浸式工作台共享同一份会话上下文
  - 保持现有 `/visualize/:sessionId` 路由作为主入口

- Non-Goals:
  - 不在本次变更中实现生产级 OpenClaw 真实适配层
  - 不在本次变更中实现复杂的多窗口或多工作台并发管理
  - 不在本次变更中实现侧栏与工作台的拖拽分栏
  - 不在本次变更中引入“新窗口打开”作为主交互

## Decisions

### Decision: 将侧栏降级为摘要视图

侧栏只负责：

- 展示当前会话的执行状态摘要
- 展示接入模式和最近更新时间
- 提供“进入沉浸式执行状态”入口

侧栏不再负责：

- 承载真实 `Star-Office-UI` iframe
- 解决完整办公室页面的显示问题

原因：

- 摘要信息适合窄侧栏
- 真实场景页适合独立工作台

### Decision: `/visualize/:sessionId` 升级为沉浸式工作台

沉浸式工作台负责：

- 承载真实 `Star-Office-UI` iframe
- 提供返回、刷新、状态展示等工具栏
- 提供更大的可视区域，减少裁切

默认路径仍为：

- `/visualize/:sessionId`

原因：

- 现有路由已经存在
- 用户语义自然
- 改造成本低于新开一路完全独立路由

### Decision: 关闭工作台时不销毁 iframe，而是隐藏保活

核心要求：

- 首次进入某个会话的沉浸式工作台时，创建该会话对应的 iframe 容器
- 当用户关闭工作台时，不卸载 iframe，只隐藏工作台容器
- 再次进入同一会话工作台时，直接恢复已存在的 iframe

原因：

- 避免重复初始化真实 `Star-Office-UI`
- 避免重复加载大体积资源
- 提升频繁开关时的响应速度

### Decision: 保活以“当前会话最近一次工作台实例”为粒度

本次变更不做复杂缓存池，只做最小可用策略：

- 至少保留当前激活会话的工作台 iframe
- 当用户切换到其他会话进入工作台时，可以替换为新的会话实例

这是一个务实选择：

- 可先满足“关闭再开不重建”的核心需求
- 避免一开始就维护大量隐藏 iframe

后续若有需要，可扩展为多会话缓存池。

## Risks / Trade-offs

- 风险：隐藏不销毁会增加前端内存占用
  - 缓解：当前阶段先只保活最近一次工作台实例，而不是无限缓存

- 风险：工作台与聊天页状态同步可能出现错位
  - 缓解：统一由 `useVisualizeStore` 管理当前激活会话和工作台可见状态

- 风险：用户可能误解“关闭”和“返回聊天”的语义
  - 缓解：按钮文案区分为“返回聊天”和“收起工作台”

## Migration Plan

1. 新增 OpenSpec 变更并确认方案
2. 调整 `useVisualizeStore`：
   - 区分摘要侧栏可见性与工作台可见性
   - 增加工作台保活状态
3. 将当前 `VisualizePanel` 改造成摘要侧栏
4. 将 `VisualizePage` 升级为沉浸式工作台
5. 抽离真实 iframe 承接组件，支持保活复用
6. 补充单元测试与页面测试
7. 回写阶段六文档

## Open Questions

- 后续是否需要支持多个会话工作台并发保活
- 是否需要在沉浸式工作台中加入“适应窗口 / 100% 缩放”控制
