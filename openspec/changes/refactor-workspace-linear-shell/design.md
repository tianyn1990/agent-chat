# 设计文档：线性分隔工作台框架重构

## 1. Context

当前主应用已经具备较稳定的 `Paper Ops / Graphite Console` 视觉基础，但框架层仍保留来自上一阶段的“壳层过渡结构”：

- 左侧 sidebar 是一整块独立圆角矩形
- 右侧 content 是另一整块独立圆角矩形
- 展开态二级 panel 依附在 sidebar 这块外壳内部

这种做法在从早期蓝白后台风迁移到 `Graphite Console` 时非常有效，因为它快速建立了统一的深色壳层语言；但随着组件与子页面逐步统一，外层“双壳层”开始变成新的限制：

- 它会持续吞掉主舞台宽度
- 它会让视觉焦点停留在“框架盒子”而不是“内容工作台”
- 它会让 rail-first 导航逻辑被旧的宽侧栏壳层惯性拖住

因此，这次不是常规的颜色或圆角微调，而是一次“框架层级改造”。

## 2. Goals / Non-Goals

### Goals

- 将应用框架改为“单一工作台底板 + 线性分隔”的结构
- 移除左右双大壳层，释放横向展示空间
- 保留内容级主舞台，使聊天阅读区、技能工具面仍然具备明确聚焦点
- 让侧边栏 rail、展开 panel、主内容区形成连续关系
- 把这套框架规则沉淀到长期设计文档和协作约束中

### Non-Goals

- 不改变信息架构本身，例如导航项、会话模型、技能功能与工作台路由
- 不把所有局部容器都删掉
- 不将沉浸式工作台重新变成说明层驱动
- 不在本次变更中引入新的第三方布局系统

## 3. 核心设计判断

### Decision: 删除“框架级壳”，保留“内容级台”

本次最重要的判断是：

- 应删掉的是应用外层的完整大壳体
- 不应删掉的是内容阅读、结果承载、工作流沉浸层等真正承担认知分组的主舞台

也就是说：

- `MainLayout.content` 不再是一块完整的圆角大卡片
- `Sidebar.sidebar` 不再是一块完整的圆角大卡片
- 但聊天页的 `messageArea`、技能页的局部工具块、结果卡片、沉浸式工作台控件，仍需保留自己的内容级语义

### Decision: 框架层级由“面”改为“线”

框架层的边界应主要通过以下手段建立：

- 纵向分隔线
- 局部浅表面层
- 更稳定的横向与纵向节奏
- 主次区域的对齐关系

而不是依赖：

- 外层大圆角
- 大面积独立背景块
- 外层阴影

### Decision: 线性分隔不等于完全扁平无层级

本次改造不是要把整个页面做成一张平面。

必须保留：

- rail 与内容区之间的强结构边界
- 内容主舞台与附属工具之间的层级差
- 结果型组件的容器语义

因此，框架级“去壳”与内容级“保台”必须同时成立。

## 4. 备选方案对比

### 方案 A：单一工作台底板 + 线性分隔 + 内容级主舞台保留

这是推荐方案。

特点：

- 移除左右整体壳层
- 用分隔线与局部弱表面层维持结构
- 聊天主舞台、技能卡、结果区保留轻量容器语义

优点：

- 空间利用率更高
- 主舞台更突出
- 风险可控，不会丢掉层级

缺点：

- 需要联动多个页面与组件适配

### 方案 B：仅缩小圆角、边框和 padding，不改框架结构

这是最省事的方案，但不推荐。

优点：

- 实现成本最低

缺点：

- 只能做局部 polish
- 不能解决“双大壳层”的根因
- 收益有限，后续仍会继续遇到类似问题

### 方案 C：完全取消所有大部分容器，做极端扁平化

不推荐。

优点：

- 理论上最省空间

缺点：

- 主次边界显著下降
- 聊天、技能、空状态、浮层的认知组织容易崩掉
- 与当前项目“专业工作台”的长期方向不一致

## 5. 模块级方案

### 5.1 主布局 MainLayout

目标：

- 从“双卡片并排”切换到“连续工作台”

设计要点：

- 外层 `layout` 继续承担整页边距与安全区
- `content` 不再作为独立大壳，而只承担右侧内容容器的弹性布局职责
- 左右区域主要通过 `gap` 与分隔线建立边界

### 5.2 Sidebar / Rail / Panel

目标：

- 让左侧成为真正的 navigation rail，而不是传统深色宽侧栏

设计要点：

- sidebar 外层去壳
- rail 保留稳定宽度和自己的内部节奏
- rail 与 panel 之间通过竖线建立一级分隔
- panel 使用极浅表面层辅助承接内容，但不再作为完整卡片壳
- 用户区、导航区、展开区都服从同一条左侧结构轨道

### 5.3 Chat Page

目标：

- 让聊天主舞台真正贴合统一工作台结构

设计要点：

- 页面整体不再依赖右侧大壳层带来的边界感
- 顶部 stage header 继续保持低存在感
- `messageArea` 保留为内容级主舞台，承接阅读轨道、滚动、底部定位按钮等关键交互
- 输入 dock 继续贴近主舞台，不再被额外外壳包裹

### 5.4 Skills Page

目标：

- 从“页面内卡片区块”切换到“工作台内工具面”

设计要点：

- 顶部 header 和 toolbar 保持单层结构
- 内容区附着在统一工作台中
- 局部统计块、空状态、卡片列表继续保留轻量容器，但不能重新长成厚页面壳

### 5.5 全局反馈组件

需要联动检查：

- `Drawer`
- `Modal`
- `Tooltip`
- `Dropdown`
- `Toast / Notification`
- `PageLoading`
- `RouteErrorBoundary`

原因：

- 一旦主框架去壳，这些浮层的颜色、描边、边距与落点会更显眼
- 如果仍保留旧的页面壳体假设，会出现风格断层或对比失衡

## 6. Token 设计建议

建议新增或重组一组“框架级 token”，避免后续又回到局部组件各自手写：

- `@frame-divider`
- `@frame-divider-strong`
- `@frame-surface-subtle`
- `@frame-surface-elevated`
- `@frame-padding`
- `@frame-gap`

用途：

- 区分框架分隔与内容容器分隔
- 让主布局、侧边栏、二级 panel 能共享同一套底层语言

## 7. 交互与可用性风险

### 风险 1：用户误以为侧边栏层级变弱

缓解：

- 保留 rail 的明确边界
- 在 active、hover、focus-visible 上强化轨道感
- 让展开 panel 与当前导航项之间存在更清晰的附着关系

### 风险 2：聊天主区在去壳后显得“散”

缓解：

- 保留 `messageArea` 主舞台
- 控制聊天列宽、内部边距和阅读轨道
- 让底部 dock 与消息区形成稳定上下关系

### 风险 3：技能页或其他页面出现不一致

缓解：

- 在实现阶段将技能页列为同步适配范围
- 将 loading、error、modal 作为全局回归检查项
- 更新 `docs/UI-视觉系统指南.md` 与 `AGENTS.md` 引用说明

## 8. 实施顺序建议

1. 先调整 token 与 `MainLayout / Sidebar` 框架层
2. 再适配聊天页，使主舞台在新框架下稳定成立
3. 再适配技能页及相关局部工具层
4. 最后做全局反馈组件与文档收敛

## 9. 成功判据

- 用户第一眼更容易感知“一个连续工作台”，而不是两个大矩形
- 侧边栏展开态不再像“卡片里套卡片”
- 主舞台横向空间可感知提升
- 页面层级没有因去壳而塌陷
- 文档可明确指导后续页面延续同一框架语言
