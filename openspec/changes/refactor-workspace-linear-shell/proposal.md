# 变更提案：将主应用框架重构为线性分隔工作台

> Change ID: `refactor-workspace-linear-shell`
> 状态：提案中
> 创建日期：2026-03-25

## 1. 背景

当前主应用已经完成 `Paper Ops / Graphite Console` 阶段的视觉收紧，但经过最新一轮实际体验和针对布局结构的复盘，可以确认页面框架仍然保留了较强的“双卡片壳层”特征：

- 左侧导航整体仍被包裹在一个完整的深色圆角矩形中
- 右侧主内容区也被包裹在另一个完整的深色圆角矩形中
- 侧边栏展开后二级面板继续附着在这个外层矩形之内
- 聊天主舞台虽然已经更克制，但外层壳层依然占据了一部分横向可视面积和视觉注意力

这种结构在早期从“默认后台风”过渡到“Graphite Console”时是有价值的，因为它帮助系统快速建立了统一的深色壳层语言；但在当前阶段，它开始暴露出新的问题：

1. 主舞台横向空间被外层双壳层继续侵占，空间利用率还有提升空间
2. 整体观感仍偏“两个并列矩形区域”，没有形成一个连续的工作台
3. 侧边栏 rail 与展开面板之间的关系还不够像“工作台内分区”，而更像“卡片中的卡片”
4. 如果继续单纯压缩圆角、阴影和 padding，收益会越来越小，因为根因是框架层级，而不是单个组件 token

用户已明确提出新的方向：

- 左右两块以及侧边栏展开的二级侧边栏，是否可以不再各自画一个大矩形
- 是否可以通过线条、分隔与连续底板来替代大面积外壳
- 是否能在保持交互质量的前提下，进一步提高空间利用率与信息密度

因此，本次 change 的目标是：在保持 `Paper Ops / Graphite Console` 产品识别的前提下，将主应用的框架层升级为：

> 单一工作台底板 + 线性分隔结构 + 内容主舞台优先

## 2. 问题定义

本次要解决的并不是“圆角是否好看”这类表面问题，而是以下结构性问题：

### 2.1 双卡片框架带来的空间浪费

当前外层存在：

- `layout` 外边距
- 左侧整体壳层边框与圆角
- 右侧整体壳层边框与圆角
- 内层页面自身的舞台容器

这使得应用同时拥有“外层框架壳”和“内层内容壳”，主舞台宽度被反复切分。

### 2.2 主舞台优先级仍可继续提升

聊天页和技能页现在已经具备较清晰的内容层级，但用户第一眼仍容易感知到“左右两块大的面板”，而不是“一个连续工作台中的主操作区域”。

### 2.3 侧边栏展开态缺少真正的工作台分区语义

侧边栏当前的 rail、二级 panel、底部用户区都处于同一个厚外壳之内。虽然视觉已较克制，但分区关系更像“侧栏卡片内分栏”，而不是“工作台底板上的导航轨”。

### 2.4 不能简单粗暴地完全去容器

如果只把大圆角矩形全部删掉，而不重新设计主舞台、分隔线和局部表面层，会带来新的问题：

- 页面会塌成一整片深色背景
- 主次区域边界会变弱
- 聊天阅读轨道可能失去聚焦
- 技能页、空状态、浮层类组件会出现风格断层

所以，本次变更必须回答两个问题：

1. 哪些“外层壳”应该删掉
2. 哪些“内容舞台”必须保留

## 3. 变更目标

### 3.1 视觉目标

- 将主应用从“左右双卡片工作区”升级为“单一连续工作台”
- 让侧边栏、展开面板和主内容区通过线性分隔建立关系
- 进一步提升空间利用率和信息密度
- 保持当前 `Graphite Console` 的深色、克制、专业气质

### 3.2 交互目标

- 让用户优先感知主舞台，而不是感知外层容器
- 保持 rail-first 导航逻辑不倒退
- 确保展开态二级侧栏仍然清晰可发现、可使用
- 保持聊天页主舞台、技能页工具层、沉浸式工作台之间的结构一致性

### 3.3 工程目标

- 将框架级分隔语言沉淀为可复用 token 和布局规则
- 避免未来新增页面又回到“单独画一个大卡片”的路径
- 为后续模块开发提供更清晰的长期 UI 约束

## 4. 方案范围

### 4.1 本次纳入范围

- 主布局外层壳层的重构策略
- 左侧 rail 与展开 panel 的线性分隔策略
- 右侧内容区从独立大壳层过渡为连续工作台的方案
- 聊天页与技能页在新框架下的适配原则
- 设计 token 与视觉系统文档的同步更新要求

### 4.2 本次不纳入范围

- 会话模型、消息协议与 WebSocket 行为
- Star-Office sidecar 或 mock 协议
- 沉浸式工作台的路由与 iframe 承接机制
- 新增大型动画系统或新的 UI 基础库

## 5. 核心方案摘要

推荐将本次变更定义为：

> 删除框架级“双大壳层”，保留内容级“主舞台容器”。

具体来说：

1. 移除左侧整体 sidebar 外壳与右侧整体 content 外壳
2. 保留一个统一的 graphite 工作台底板，由页面背景与布局结构共同承接
3. 使用垂直分隔线、局部浅表面层和稳定间距，替代原有大圆角矩形边界
4. 聊天页继续保留消息主舞台，但其外层不再依赖另一层完整壳体
5. 技能页改为真正附着在统一工作台上的工具面，而不是“卡片化页面”
6. 侧边栏展开 panel 改成 rail 附着式分区，而不是嵌套在大卡片里的第二块卡片

## 6. 预期收益

### 6.1 空间收益

- 横向可视宽度增加
- 主舞台有效展示面积提升
- 侧边栏展开时的空间挤压感降低

### 6.2 视觉收益

- 页面更像连续工作台，而不是左右两个独立盒子
- 外层 chrome 进一步变薄
- 信息层级更依赖结构、分隔与密度，而不是依赖大面积壳层

### 6.3 长期收益

- 为后续页面提供明确的框架级模式
- 降低局部页面随意画大卡片导致风格漂移的风险
- 更符合当前产品“主界面克制专业、沉浸层单独表达”的边界

## 7. 风险与异议

### 7.1 风险：主内容与侧栏边界变弱

如果只删除外壳，不补足分隔规则，页面会失去结构边界。

应对策略：

- 明确新增框架分隔 token
- 保留 rail 与主内容之间的强分隔线
- 保留聊天页主舞台等内容级关键容器

### 7.2 风险：二级侧栏展开态失去附着感

如果只是去背景，panel 会看起来像“裸露内容块”。

应对策略：

- 将 rail 与 panel 设计成同一工作台中的相邻分区
- 使用内部分隔与局部弱表面层而不是完整卡片壳

### 7.3 风险：其余模块风格断层

如果只改主框架，不改聊天页和技能页内部的结构承接，整体会不协调。

应对策略：

- 将聊天页、技能页列为本次必须适配模块
- 将登录页、错误页、loading、drawer、modal 作为联动检查项

## 8. 成功标准

本次 change 评审通过的标准：

- 能清晰定义“框架级壳层”和“内容级舞台”的边界
- 能清晰说明为什么不是简单删圆角，而是重构为线性分隔工作台
- 能明确指出哪些模块必须跟着适配
- 能给出长期可维护的 token / 布局 / 文档沉淀策略
- 变更方案可作为后续实现阶段的直接输入
