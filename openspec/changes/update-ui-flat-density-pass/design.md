# 设计文档：收紧主应用视觉密度并优化沉浸式工作台控件

## Context

当前系统已经具备统一的 `Paper Ops / 智能工作台` 视觉主轴，但经过一轮真实页面观察后，可以确认现阶段最主要的问题已经不再是“风格有没有建立”，而是“风格是否足够适合作为高频工作台长期使用”。

现状中的几个明显症状：

- 大量区域依赖阴影、渐变和大圆角建立层级，导致主应用偏“软”
- 标题、hero、统计卡与建议卡的视觉尺度偏大，导致页面偏“松”
- 消息区与输入区偏向展示型大卡片，不够像连续作业面
- 沉浸式工作台外层工具条虽然已经极简化，但当前位置与形态仍有“看不见 / 容易遮挡 / 与 iframe 内控件打架”的问题

用户给出的参考图并不是要求系统完全转为黑色极简风，而是希望吸收其中更高信息密度、更薄的 chrome、更少层叠视觉修饰的优点。

因此本次设计的核心不是换风格，而是：

> 在保留 `Paper Ops` 产品识别度的前提下，将主应用进一步收紧为 `Flat Console` 化的操作界面。

## Goals / Non-Goals

### Goals

- 降低常规页面对阴影与厚卡片的依赖，改用边框、分隔与明度差建立层级
- 提高聊天页、技能页、登录页的信息密度，减少“宣传感”和“展板感”
- 保持暖白主画布和深色侧栏的产品识别，但让内容结构更利落
- 优化沉浸式工作台工具条的可发现性、占位策略和遮挡边界
- 统一关键页面和关键操作的图标语言
- 将这一轮调整明确写入 OpenSpec 与视觉系统文档，避免后续回退

### Non-Goals

- 不将主应用整体改造成参考图中的纯黑极简产品
- 不改变当前功能路径、会话模型、Mock 逻辑或 `Star-Office-UI` 接入协议
- 不重写聊天、技能、登录等页面的信息架构
- 不在本次 change 中引入重量级设计系统框架或额外视觉运行时依赖

## Decisions

### Decision: 视觉方向升级为 `Paper Ops / Flat Console`

保留现有暖纸面主背景、稳重靛蓝主色和铜棕辅助色，但减少“柔和卡片”气质，强调：

- 更薄的表面层
- 更紧凑的间距系统
- 更小的圆角体系
- 更低的阴影存在感
- 更强的分隔、对齐和信息密度

这样可以保持已有品牌识别，而不回退到通用蓝白后台风，也不被参考图完全牵引到另一套气质。

### Decision: 常规层级主要依赖边框与表面差，不依赖阴影

本次调整中：

- 常规容器默认使用轻边框 + 轻明度差
- 阴影只保留给少数浮层、悬浮条和覆盖层
- 页面区块不再普遍使用大投影

这会让页面更扁平，也更适合高密度信息场景。

### Decision: 聊天页从“双大卡片”收紧为连续工作面

聊天主舞台的优先级最高，本次要把：

- 顶部标题区从“标题 + 两张厚 meta 卡”调整为更紧凑的工作台栏
- 消息区和输入区从两个厚盒子，收敛为同一工作面上的上下两个功能层
- 欢迎态从“宣传型 hero + 大建议卡”调整为“首屏可完成行动”的工作台起始页

### Decision: 技能页与登录页都要降低展示感

技能页要像能力管理工作面，而不是零售型卡片列表；登录页要像产品入口，而不是品牌海报页。

对应策略：

- 技能页减少统计卡体积、收紧工具栏、提升技能卡信息密度
- 登录页降低左侧 hero 压迫感，强化右侧登录入口的操作优先级

### Decision: 沉浸式工作台使用“边缘工具坞”而不是“悬浮大胶囊”

当前沉浸式工作台中，问题不只是按钮大小，而是工具条定位策略不合理。

本次明确：

- 不再让工具条漂浮在像素办公室核心可视区域
- 工具控件优先沿外层边缘布置
- 主操作和次操作分层
- 保持 `iframe-first`，不得用新说明面板重新挤占办公室画面

### Decision: 图标体系统一到更薄、更中性的 line icon

导航、动作与状态图标应尽量统一到相同 stroke 风格与尺寸体系，减少当前部分图标“过粗、过重、过满”的问题。

## Module Plan

### 1. 全局 token

- 下调圆角档位
- 收紧间距变量
- 阴影减少为更克制的两到三档
- 减少大面积 radial / glow 的使用频率

### 2. 聊天页

- 收紧 `stageHeader`
- 将 meta 信息改成紧凑摘要样式
- 收紧 `messageArea` / `inputArea`
- 压缩欢迎态 hero 和建议卡高度
- 建立更稳定的消息阅读轨道和时间戳附属层

### 3. 技能页

- 统计区转为紧凑 summary
- 搜索与分类工具栏更清晰
- 技能卡正文密度提高
- 空状态收紧高度和占位

### 4. 登录页

- 降低左侧 hero 体量
- 收紧标题与正文
- 将视觉重心转向右侧 CTA 卡片

### 5. 沉浸式工作台

- 重新定位工作台上层工具控件
- 以边缘窄工具坞承载返回、刷新、收起等操作
- 工具条需要更清晰但更少遮挡

## Risks / Trade-offs

- 过度扁平化可能导致页面失去已有的温度和品牌感
  - 通过保留暖白纸面、深色侧栏和局部铜棕强调来平衡

- 过度压缩信息密度可能影响可读性
  - 通过只压缩结构性空白，不压缩正文行高与点击热区来控制

- 沉浸式工作台工具条若过于隐蔽，可能降低可发现性
  - 通过“主操作更明确、次操作收纳”的方式解决，而不是重新做大悬浮层

- 图标体系切换可能带来局部风格混杂
  - 通过优先替换导航、主操作和工作台 chrome，分批完成统一

## Migration Plan

1. 创建并校验本次 OpenSpec change
2. 收紧全局 token 与基础容器语言
3. 优先改造聊天页与工作台控件
4. 再改造技能页与登录页
5. 同步更新测试与 `docs/UI-视觉系统指南.md`
