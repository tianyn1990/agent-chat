## MODIFIED Requirements

### Requirement: 系统必须提供统一的产品级视觉系统

系统 MUST 在主应用中提供统一的视觉系统，覆盖颜色、字体、容器、状态色、背景层次与交互反馈，并在主要页面间保持一致；该视觉系统还 MUST 兼顾扁平化表达、较高信息密度和长期使用时的操作效率，不得重新回到过度依赖厚卡片、大阴影和大圆角的展示型界面。

#### Scenario: 主页面共享统一且收紧后的视觉 token

- **Given** 用户分别访问登录页、聊天页、技能页与执行状态工作台
- **When** 页面完成渲染
- **Then** 页面必须共享统一的品牌主色、基础字体、边框语义和状态反馈语言
- **And** 常规容器层级应优先通过边框、明度差和布局建立
- **And** 不得让主要页面重新出现厚重阴影、大块渐变和过大圆角主导层级的情况

### Requirement: 聊天页必须作为高频使用的主工作台呈现

系统 MUST 将聊天页呈现为清晰、紧凑、可连续使用的主工作台界面，而不是由多个展示型大卡片拼接而成的聊天布局。

#### Scenario: 用户进入空会话

- **Given** 用户进入一个没有消息的新会话
- **When** 欢迎态出现
- **Then** 首屏必须能够完整呈现主要欢迎信息、建议入口和输入区域
- **And** 欢迎态必须避免使用过厚的 hero 与过高的建议卡导致首屏内容被内部滚动截断

#### Scenario: 用户查看已有会话

- **Given** 用户进入一个已有消息的会话
- **When** 消息列表、标题摘要与输入区同时可见
- **Then** 标题区、消息区和输入区必须形成连续工作面
- **And** 标题 meta 必须以紧凑摘要形式呈现，而不是与主标题争夺视觉带宽

### Requirement: 消息与结果组件必须支持高密度阅读

系统 MUST 对消息气泡、图表结果、卡片结果、文件消息、时间信息和消息操作建立稳定的阅读轨道，以支持更高密度的连续阅读。

#### Scenario: 用户浏览长对话

- **Given** 同一会话中存在多条用户消息与助手消息
- **When** 用户滚动浏览对话
- **Then** 用户消息与助手消息必须形成稳定可预测的左右阅读关系
- **And** 时间戳、操作按钮和附属信息必须贴近所属消息收纳
- **And** 不得让附属信息像散落注释一样破坏连续阅读

#### Scenario: 助手返回不同类型结果

- **Given** 助手在同一会话中返回文本、图表、卡片或文件消息
- **When** 用户查看这些结果
- **Then** 结果容器必须延续统一且更扁平的工作台语言
- **And** 不得让复杂结果继续表现为厚重漂浮卡片

### Requirement: 技能页与登录页必须降低展示感并提高操作聚焦

系统 MUST 将技能页和登录页调整为更偏操作型的页面：信息区更紧凑，主操作更明确，页面不再以大面积展示型区块主导。

#### Scenario: 用户查看技能页

- **Given** 用户进入技能页
- **When** 页面渲染技能统计、搜索筛选和技能卡
- **Then** 搜索与筛选必须先于装饰性统计获得视觉优先级
- **And** 技能卡必须提高正文信息密度
- **And** 空状态与统计区不得继续占用过高垂直空间

#### Scenario: 用户进入登录页

- **Given** 用户尚未登录
- **When** 登录页加载完成
- **Then** 页面必须优先突出登录 CTA 区
- **And** 左侧品牌区必须控制视觉体量，避免其压过主登录动作

### Requirement: 图标系统必须保持统一的线性视觉语义

系统 MUST 在导航、主操作、状态提示和工作台控件中保持统一的图标风格、尺寸语义和视觉重量，不得长期混用风格差异明显的图标表达。

#### Scenario: 用户在多个页面浏览主操作

- **Given** 用户依次浏览侧边导航、聊天操作、技能操作和登录入口
- **When** 用户观察各处图标
- **Then** 图标应具有统一的 stroke 风格和相近的视觉重量
- **And** 不得出现局部图标明显更厚重或更装饰化而破坏整体一致性
