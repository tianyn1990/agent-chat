# 设计文档：Graphite Console 工作台重构

## Context

在 `Paper Ops / Flat Console` 完成后，系统已经摆脱了早期的厚卡片与展示型视觉，但最新一轮截图和参考图进一步暴露出一个问题：

- 我们优化了“组件”，但还没有真正重构“应用壳层”

参考图的关键优势并不是某个圆角、某个阴影或某个 icon，而是整个应用具备一套更接近原生 AI 产品的视觉与交互秩序：

1. 外层 chrome 很薄
2. 左侧更像 navigation rail，而不是传统业务侧栏
3. 欢迎态的主舞台极简，输入条是底部 dock
4. 进入对话后，内容成为第一主角，而不是继续被容器和说明包裹
5. 技能页与聊天页属于同一套 dark console，而不是两个不同风格的页面
6. 沉浸式工作台的外层控件极简到“只解决返回与极少数操作”

因此，本次不是简单的暗色化，而是一次带结构调整的壳层升级。

## Goals / Non-Goals

### Goals

- 将主应用从“收紧后的业务工作台”升级为“深色原生 console 工作台”
- 让聊天页真正成为主舞台，头部说明与附属元信息显著退后
- 让欢迎态与对话态形成统一但不同的两套舞台结构
- 让输入条成为稳定、明确的底部 dock，而不是独立厚表单卡
- 提升技能页的信息密度与工具感，减少市场页、展示页气质
- 保持沉浸式 `Star-Office-UI` 为独立沉浸层，不让主应用 chrome 重新覆盖真实像素办公室
- 将本轮视觉与交互原则沉淀到长期文档，约束后续新增模块

### Non-Goals

- 不改动会话模型、消息协议、Mock 机制或 Star-Office sidecar 协议
- 不把沉浸式工作台改回侧边栏嵌入模式
- 不引入新的大型 UI 库或重型动画运行时
- 不把整站做成高度炫技或强装饰的视觉实验品

## Decisions

### Decision: 主应用升级为 `Paper Ops / Graphite Console`

本次不丢掉现有 `Paper Ops` 产品识别，而是把“暖纸面工作台”的表达压低，将视觉重心切到：

- 深 graphite 背景
- 更薄的边框与分隔
- 更小的圆角
- 更低存在感的 hover / shadow
- 更强的布局秩序与阅读轨道

也就是说：

- 保留专业、克制和工作台属性
- 放弃偏暖、偏厚、偏组件化的观感

### Decision: 左侧结构从宽侧栏转向 rail-first

参考图的一个核心启发是：

- 用户首先应该感知“主舞台”
- 而不是感知“侧栏占了很多空间”

因此本次建议把左侧壳层调整为：

- 更窄的主 rail，承载一级导航与用户入口
- 当前页局部需要的二级内容，再在页面内部展开

对于聊天页，这意味着会话列表不再天然占据一个大宽栏，而要以更轻的形态出现。

### Decision: 聊天页重构为“中心舞台 + 底部 dock”

聊天页将从“标题区 + 消息容器 + 输入容器”的后台结构，调整为：

- 欢迎态：中心舞台，标题与入口居中收敛，建议入口更像 quick actions
- 对话态：正文阅读轨道前置，头部说明最小化
- 底部：composer 作为 dock，稳定停靠底部，不再像独立卡片

### Decision: 普通文本消息弱容器化

当前文本消息仍带有较明显 bubble / card 语义。参考图更接近：

- 用户消息：轻量右侧提示气泡
- 助手文本：文本优先，必要时只有非常轻的表面层
- 结果型内容：卡片、图表、文件结果继续保持明确容器

因此本次将区分：

- narrative message：弱容器
- result block：强容器

### Decision: 技能页改造成能力控制面

技能页不能继续停留在“卡片市场”思路上。本次将明确把它改成：

- 顶部工具栏单层承载标题、搜索、创建
- 分类筛选压缩为更轻 segmented controls
- 卡片更像能力模块条目，减少大面积留白和装饰

### Decision: 沉浸式工作台外层控件进一步降噪

当前工作台控件已经比之前克制，但参考图的启发更明确：

- 这类沉浸层外壳只应该保留“必要导航”
- 任何大块悬浮说明、元信息、说明文字都会破坏沉浸感

因此本次继续坚持：

- `iframe-first`
- 关闭即隐藏、不卸载
- 返回按钮清晰但极简
- 次级操作更低存在感

## Module Plan

### 1. 全局 token 与壳层

- 将主背景与面板体系切到更深的 graphite / ink 系列
- 重新定义 sidebar / surface / border / hover token
- 收紧布局圆角、边框和分隔表达

### 2. 主布局

- 调整侧栏为更窄的 rail-first 结构
- 收紧 logo、导航、底部用户区的节奏与尺寸
- 让内容区拥有更完整的主舞台宽度

### 3. 聊天页

- 头部摘要极小化，仅保留必要上下文
- 欢迎态改为中心式舞台布局
- 建议入口改为轻量 chips + 紧凑模块入口
- 对话态改为更自由的阅读画布
- 输入区重构为底部 dock composer

### 4. 消息与结果

- 普通文本消息弱容器化
- 用户消息体量继续减小
- 结果型卡片、图表、文件结果保留明确容器
- 消息操作浮现策略更轻

### 5. 技能页与登录页

- 技能页压缩头部与工具栏，重排卡片密度
- 登录页同步 dark console 壳层语言，避免风格断层

### 6. 沉浸式工作台

- 工具条进一步靠边、减小体积
- 返回路径固定且易发现
- 避免任何重新挤占 iframe 画面的信息层

## Risks / Trade-offs

- 如果壳层切换过快，可能与已有暖纸面识别产生冲突
  - 通过保留少量 `Paper Ops` 品牌色作为 accent，而不是完全改成纯黑极简

- rail-first 结构可能让部分用户觉得会话列表不够显眼
  - 通过在聊天页内部提供更轻但可发现的会话入口补偿，而不是回到宽侧栏

- 文本消息过度弱容器化可能影响阅读边界
  - 通过阅读轨道宽度、行高、间距和 hover 反馈维持可读性

- 沉浸式工作台控件如果收得太狠，可能影响新用户发现
  - 保留一个明确的返回主操作，其余操作弱化但不消失

## Migration Plan

1. 创建并校验 `update-workspace-graphite-console` change
2. 先改 token 与主布局壳层
3. 重构聊天页欢迎态、对话态和底部 dock
4. 调整消息/结果容器层级
5. 重构技能页与登录页壳层
6. 收敛沉浸式工作台控件
7. 更新测试与 `docs/UI-视觉系统指南.md`
