# 变更提案：补充 Star-Office-UI 本地 Mock 能力

> Change ID: `add-local-star-office-mock`
> 状态：已实现，待归档
> 创建日期：2026-03-24

## 1. 背景

当前阶段六已经完成了以下前置工作：

- 前端已具备“查看执行状态”的入口与面板承接能力
- 前端已引入 `sessionId` 维度的运行态缓存
- 文档层已明确 Star-Office-UI 本质上是执行状态看板，而不是工作流 DAG
- 服务端适配层方案已经完成设计，但尚未接入真实 OpenClaw / Star-Office-UI 服务

但在本地开发阶段，仍然存在一个直接阻碍调试的问题：

- 用户执行 `npm run dev` 后，虽然能看到对话功能与 Mock WebSocket 回复
- 但无法真实看到 Star-Office-UI 在 iframe 中的展示效果
- 因为本地没有运行中的 Star-Office-UI 服务，也没有向其提供 `/status`、`/agents` 等接口的适配层

这导致：

- 前端执行状态面板只能停留在“未配置地址”或空白 iframe 层面
- UI、交互、会话绑定、多会话切换、状态流转都无法在本地自洽验证
- 后续接入真实服务端时，很多问题会被推迟到联调阶段集中暴露

## 2. 问题定义

本次变更要解决的问题不是“本地复刻完整 Star-Office-UI 项目”，而是：

- 在 `npm run dev` 单进程开发模式下
- 提供一个同域、可嵌入、可按会话区分的本地 mock 执行状态页面
- 同时提供一层本地 mock adapter，模拟 Star-Office-UI 所需接口与状态来源

这样可以让本地开发验证以下关键链路：

1. 从 AI 消息点击“查看执行状态”
2. 面板打开并绑定当前 `sessionId`
3. iframe 加载本地 mock 页面
4. mock 页面按当前 `sessionId` 拉取状态
5. 当聊天运行态变化时，iframe 中的执行状态同步变化

## 3. 变更目标

### 3.1 功能目标

- 在本地开发模式下可直接看到 iframe 中的执行状态页面
- 页面内容与当前选中会话强绑定，而不是全局共享单一状态
- mock adapter 接口尽量贴近 Star-Office-UI 的真实接口模型
- 保留“主应用 -> 适配层 -> 状态看板”的真实部署拓扑，避免未来切换到真实服务时重做前端接线

### 3.2 工程目标

- 只在开发模式启用，不污染生产构建
- 尽量复用现有 `Mock WebSocket` 和 `useVisualizeStore` 的运行态
- 不引入额外的本地外部服务进程，最好保持 `npm run dev` 即可运行
- 为后续真实服务端适配层接入预留清晰切换点

### 3.3 文档目标

- 给实现前评审提供完整、统一的方案入口
- 明确本地 mock 与未来真实适配层之间的边界
- 明确多会话场景下“当前到底展示哪个会话”的规则

## 4. 范围

### 4.1 本次纳入范围

- 本地 mock adapter 的职责、接口与数据模型设计
- 本地 mock 页面在 iframe 中的承接方式设计
- `sessionId` 维度的状态隔离与多会话语义设计
- 开发环境的配置开关与 URL 路由策略设计
- 与现有 Mock WebSocket / 前端运行态的桥接设计

### 4.2 本次不纳入范围

- 真实 Star-Office-UI 项目的二次开发
- 真实 OpenClaw 服务端适配层实现
- 生产环境部署脚本与容器配置落地
- Pixel 风办公室素材的 1:1 完全复刻
- 真正的 workflow DAG、trace、节点图可视化

## 5. 方案摘要

推荐新增一套开发期专用的本地 mock 机制：

1. 在 Vite 开发服务内提供本地 mock adapter 路由
2. 在同一个开发服务下提供可被 iframe 访问的 mock 状态页面
3. 主应用把当前会话运行态桥接到本地 mock adapter
4. mock 页面根据 `sessionId` 从 mock adapter 读取状态并展示

这套方案有四个关键优点：

- 同域：避免跨域问题，贴近未来同域 iframe 部署形态
- 单进程：无需单独启动 Python/Flask 或真实 Star-Office-UI 服务
- 会话隔离：可以验证多会话状态不串线
- 易切换：未来只需替换数据源地址，不必推翻页面承接逻辑

## 6. 为什么值得现在做

如果没有本地 mock：

- 当前可视化入口虽然已经存在，但无法完成有效验收
- 后续真实联调会同时叠加 iframe、路由、状态流、会话映射、接口契约等多类问题
- 任何“这个面板到底显示哪个会话”的语义问题都要到联调时才能发现

如果先补本地 mock：

- 可以先把前端承接与多会话语义校正到位
- 可以在没有真实 Star-Office-UI/后端适配层的情况下先完成 UI 联调
- 可以把本地 mock 作为阶段七前的稳定测试夹具保留下来

## 7. 影响范围

### 7.1 前端

- `VisualizeWorkspaceView` 的 iframe 地址生成逻辑
- 执行状态面板的本地 mock 开关判断
- Chat 页面对会话运行态的桥接逻辑

### 7.2 开发环境

- Vite 开发期路由 / 中间件
- `.env.local` / `.env.example` 的新增配置项

### 7.3 测试

- 新增本地 mock adapter 的单元测试或集成测试
- 新增 mock 页面按 `sessionId` 展示状态的测试
- 新增“多会话不串状态”的测试

### 7.4 文档

- 阶段六文档需要引用本次 OpenSpec 变更
- 后续实现完成后，需要把结果回写到阶段文档和项目总览

## 8. 风险与代价

### 8.1 风险

- 如果 mock 接口和真实 Star-Office-UI 接口差异过大，未来切换时仍会返工
- 如果 mock 页面做得过重，可能演化为维护成本高的“伪产品”
- 如果会话语义设计不严谨，反而会固化错误认知

### 8.2 控制策略

- mock adapter 只做最小必要接口，不扩展与真实系统无关的自定义协议
- 页面只验证“状态看板承接能力”，不追求像素级复刻
- 全部状态以 `sessionId` 为主键，避免使用全局单例状态

## 9. 成功标准

本次 change 评审通过的标准：

- 方案能回答“为什么本地 `npm run dev` 看不到效果”
- 方案能回答“本地 mock 展示的是哪个会话”
- 方案能回答“如何避免多会话串状态”
- 方案能定义清晰的本地接口与切换边界
- 文档足够完整，确认后可直接进入开发
