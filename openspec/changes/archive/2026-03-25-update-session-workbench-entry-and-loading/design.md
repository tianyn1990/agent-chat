# 设计文档：会话级执行状态入口与工作台冷启动优化

> 对应 Change: `update-session-workbench-entry-and-loading`

## 1. 设计结论

本次设计有四个明确结论：

1. `Star-Office-UI` 的执行状态入口必须从“消息级动作”收敛为“会话级主入口”
2. 首次进入沉浸式工作台时，不应直接暴露裸白 `iframe`，而应由外层宿主承接冷启动过程
3. 工作台缓存继续坚持“关闭仅隐藏、不主动销毁”，并进一步升级为 `LRU=2`
4. 会话级入口可以借用像素办公室的识别语义，但不能把主应用重新做成像素游戏风

## 2. 现状问题分析

## 2.1 语义错位

当前实现虽然是按 `sessionId` 打开工作台，但入口仍在消息操作区，带来以下问题：

- 用户会自然推断“这条回复对应一个独立的执行状态”
- 同一会话下每条 assistant 消息都重复出现相同入口，造成视觉噪音
- 当会话很长时，用户想回到工作台，反而要先回忆“去点哪条消息下面的按钮”

这和实际架构并不一致。当前前端与可视化 store 的真实事实是：

- runtime 存储是 `runtimeBySession`
- 工作台路由是 `/visualize/:sessionId`
- iframe URL 也是按 `sessionId` 生成

因此，继续保留消息级主入口只会扩大理解成本。

## 2.2 冷启动白屏

某个会话第一次进入沉浸式工作台时，白屏往往来自几层成本叠加：

1. 路由切换到 `/visualize/:sessionId`
2. 全局工作台宿主从隐藏态切到可见态
3. iframe 首次请求 facade / mock / real sidecar 页面
4. Star-Office 页面完成脚本初始化
5. Phaser 场景、像素资源和状态轮询开始工作

当前宿主虽然已经支持“同会话隐藏不销毁”，但对“第一次打开”并没有提供中间承接层，因此用户看到的是：

- 空背景
- 裸白 iframe
- 或短暂没有上下文的壳层

## 2.3 缓存边界

完全不保活会导致每次进入都重新初始化办公室；但无限保活多个会话的 iframe 也不合理，因为：

- `iframe + Phaser` 运行时相对重
- 多会话长期常驻会增加内存与后台活动成本
- 主应用只需要保留最近少量会话的恢复能力

因此更合适的策略是小容量受控缓存，而不是“只保一个”或“全保”。

## 3. 目标

## 3.1 Goals

- 让执行状态入口在语义上明确属于“当前会话”
- 让首次进入工作台从“白屏等待”变成“分阶段进入”
- 保持 `iframe-first`，不重新引入占视口的大说明面板
- 用低侵入方式增强入口辨识度
- 在性能与体验之间平衡，采用 `LRU=2`

## 3.2 Non-Goals

- 不修改真实 `Star-Office-UI` 的页面内容
- 不改变服务端适配层的总体架构
- 不把真实像素办公室重写成 React 组件
- 不引入多于 2 个以上的长期活跃工作台缓存

## 4. 方案总览

## 4.1 入口迁移

聊天页顶部会话头部新增固定入口，作为当前会话的“执行状态主入口”。

入口位置原则：

- 与当前会话标题、状态胶囊同一层级
- 保持接近右上 meta 区，而不是放回消息区
- 在欢迎态与已有消息态中都可见

消息动作区不再承担主入口职责：

- 默认移除“查看执行状态”
- 如未来确实需要消息级上下文跳转，也只能作为次级能力重新定义，不能与会话级入口并存为双主入口

## 4.2 像素工位徽章

会话头部入口采用“像素工位徽章”样式。

设计原则：

- 只借用像素办公室的记忆点，不复制其整体风格
- 按钮整体仍服从主应用的扁平、低阴影、细边框体系
- 左侧使用简化的像素工位 glyph，右侧显示主文案与当前运行态

状态表达：

- `idle`：静态低亮，副标为“待命”
- `running`：glyph 内部或右侧小点做轻量活动反馈，副标为“处理中”
- `error`：转为低饱和警示色
- `warming`：在点击到 iframe ready 之间显示“进入工作台”的过渡态

为什么不用真像素按钮：

- 真像素按钮会与主应用整体 chrome 冲突
- 容易让聊天页从“工作台”退化成“游戏界面”
- 双主题适配成本也会更高

## 4.3 分阶段进入

首次进入工作台时，交互分为三段：

1. 用户点击会话级入口
2. 工作台壳层立即打开，并显示极简加载封面
3. iframe 完成 `load` 后，加载封面淡出，真实办公室接管

加载封面只保留最小必要信息：

- 当前会话标题
- 执行状态标签
- 最近一条 runtime detail
- 极简“正在进入沉浸式工作台”提示

不做的内容：

- 不重新铺大面积帮助说明
- 不在工作台里塞二级摘要栏
- 不让封面长期停留占视口

## 4.4 预热策略

当前活跃会话允许静默预热。

建议行为：

- 当用户进入某个会话且该会话存在执行状态入口时，宿主可在后台预创建 iframe 容器
- 预热态 iframe 保持隐藏，不抢占焦点
- 若用户随后点击入口，优先复用已预热实例

预热边界：

- 只对当前活跃会话进行预热
- 预热失败时不弹全局错误，只在真正打开工作台时给出明确提示

## 4.5 `LRU=2` 保活策略

工作台缓存从“单一当前实例保活”扩展为“最近两会话 LRU”。

设计原因：

- 用户常见路径是 A 会话看办公室，返回，再切到 B，再回 A
- `LRU=2` 可以覆盖最常见的往返切换
- 相比更多缓存实例，`2` 在体验收益和资源占用之间更平衡

缓存规则建议：

- key 使用 `sessionId`
- 容量固定为 `2`
- 当打开第三个不同会话时，淘汰最久未使用且当前不可见的实例
- 同一会话的“刷新”操作可以显式重建该会话实例

## 4.6 路由角色收敛

`/visualize/:sessionId` 继续保留，但角色收敛为：

- 深链接入口
- 刷新恢复入口
- 宿主显隐同步信号

不再让用户感知为：

- “先跳到另一个页面”
- “再等待 iframe 出来”

因此交互主轴应是：

- 在聊天会话内点击主入口
- 立即进入沉浸式工作台
- 路由变化只是状态同步结果，不应成为用户感知中的主过程

## 4.7 返回后的轻量提示

从工作台返回聊天页后，轻量恢复提示仍然保留，但职责变更为：

- 辅助恢复入口
- 短驻留状态反馈

它不再承担主入口职责，因此其视觉权重应继续低于会话头部入口。

## 6. 风险与取舍

## 6.1 为什么不直接把多个会话都长期挂着

这样做虽然可以减少切换等待，但会带来更高资源占用，且在 `Phaser` 场景下收益不成比例。

## 6.2 为什么不继续保留消息级入口作为补充

当前用户已经明确指出它本质是会话级能力。若会话级与消息级同时并存，会再次制造：

- “到底哪个才是主入口”
- “是不是不同消息对应不同办公室状态”

因此本轮设计不建议“双入口并存”。

## 6.3 为什么不直接在聊天页内嵌一个小办公室缩略窗

这会重新把真实 iframe 压缩进主界面局部区域，违背此前已经收敛好的 `iframe-first` 边界，也会牺牲办公室本身的可视面积。

## 7. 验收标准

- 用户在聊天页能明确感知“当前会话”的执行状态入口
- 入口具有稳定识别度，但不破坏主应用视觉系统
- 首次进入工作台不再出现裸白感明显的空壳阶段
- 已打开过的最近两会话可被快速恢复
- 消息区不再承担执行状态主入口职责
- 关闭工作台后，用户仍可通过轻量提示或会话头部入口恢复
