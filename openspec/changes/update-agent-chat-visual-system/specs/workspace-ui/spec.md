## ADDED Requirements

### Requirement: 系统必须提供统一的产品级视觉系统

系统 MUST 在主应用中提供统一的视觉系统，覆盖颜色、字体、容器、状态色、背景层次与交互反馈，并在主要页面间保持一致。

#### Scenario: 主页面共享统一视觉 token

- **Given** 用户分别访问登录页、聊天页、技能页与执行状态工作台
- **When** 页面完成渲染
- **Then** 页面必须共享统一的品牌主色、基础字体、容器边界与状态反馈语言
- **And** 不得出现某一主页面仍明显保持旧版默认后台风格

### Requirement: 聊天页必须作为产品主工作台呈现

系统 MUST 将聊天页呈现为清晰的主工作台界面，而不是仅由默认侧边栏、消息区和输入框拼接而成的通用聊天布局。

#### Scenario: 用户进入空会话

- **Given** 用户进入一个没有消息的新会话
- **When** 欢迎态出现
- **Then** 页面必须展示具有明确任务入口和产品气质的欢迎界面
- **And** 欢迎界面必须与主聊天舞台属于同一视觉语言

#### Scenario: 用户查看已有会话

- **Given** 用户进入一个已有消息的会话
- **When** 消息列表、会话列表和输入区同时可见
- **Then** 聊天页必须保持清晰的信息主次关系
- **And** 会话区、消息区、输入区必须形成统一工作台结构

### Requirement: 会话与消息组件必须体现分层信息设计

系统 MUST 对会话项、消息气泡、消息操作、图表结果、卡片结果和文件消息使用一致且可区分的分层视觉语言。

#### Scenario: 助手返回不同类型消息

- **Given** 助手在同一会话中依次返回文本、图表、卡片或文件相关内容
- **When** 用户查看这些消息
- **Then** 系统必须让不同类型结果拥有清晰可辨的容器层次
- **And** 不得让所有结果都退化为相同的普通白色卡片感

#### Scenario: 用户浏览历史会话

- **Given** 用户打开左侧会话区
- **When** 用户查看当前会话、普通会话和空状态
- **Then** 系统必须通过层级、摘要和状态提示建立清晰差异

### Requirement: 技能页与登录页必须纳入同一产品视觉叙事

系统 MUST 将技能页、登录页及其相关子组件纳入与聊天页一致的产品视觉系统，而不是保留孤立的功能页样式。

#### Scenario: 用户在聊天页与技能页之间切换

- **Given** 用户从聊天页切换到技能页
- **When** 技能页加载完成
- **Then** 页面必须延续统一的产品视觉语言
- **And** 不得呈现明显割裂的另一套设计风格

#### Scenario: 用户进入登录页

- **Given** 用户尚未登录
- **When** 系统展示登录页或登录回调承接页
- **Then** 页面必须体现产品级品牌入口感
- **And** 加载与状态反馈必须与主应用保持一致语法

### Requirement: 执行状态工作台必须与主产品形成一致但分层的体验

系统 MUST 保持执行状态工作台的沉浸式 `iframe-first` 特性，同时让其外层控件、背景与过渡反馈与主应用属于同一产品体验。

#### Scenario: 用户从聊天页进入执行状态工作台

- **Given** 用户在聊天页点击查看执行状态
- **When** 沉浸式工作台打开
- **Then** 工作台必须优先展示真实执行状态 iframe
- **And** 外层悬浮控件必须保持极简
- **And** 主产品与工作台之间的切换必须具有一致的视觉语义

### Requirement: UI 升级不得破坏可用性基线

系统 MUST 在视觉升级后保持桌面场景下的可读性、交互可点击性、响应式适配和性能基线。

#### Scenario: 中等宽度桌面视口

- **Given** 用户在中等宽度桌面视口中访问聊天页或技能页
- **When** 页面完成布局
- **Then** 关键内容必须仍然可读且可操作
- **And** 不得因纯视觉改造导致核心区域严重挤压或溢出

#### Scenario: 用户使用键盘和可见焦点

- **Given** 用户通过键盘操作主要按钮、导航项或输入控件
- **When** 焦点移动到对应元素
- **Then** 系统必须提供清晰可见的焦点反馈
