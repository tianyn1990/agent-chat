# 变更提案：全面升级 agent-chat 的产品级 UI 视觉系统

> Change ID: `update-agent-chat-visual-system`
> 状态：提案中
> 创建日期：2026-03-24

## 1. 背景

当前 `agent-chat` 已经具备相对完整的产品骨架：

- 登录、对话、技能市场、执行状态工作台等主路径已经打通
- `Star-Office-UI` 真实像素办公室已经具备嵌入与沉浸式承接能力
- 多会话、卡片、图表、文件上传、Mock 与真实链路等基础设施已经具备

但从产品观感看，当前前端仍然主要停留在“功能可用”的阶段，而未形成足够鲜明的品牌化视觉语言。主要问题包括：

- 整体气质偏通用后台风，页面之间的视觉辨识度不足
- 聊天页、欢迎页、会话列表、技能页、登录页之间缺少统一的设计叙事
- 侧边栏、消息气泡、状态入口、图表容器等局部已多次迭代，但全局视觉系统尚未收敛
- 已接入的 `Star-Office-UI` 像素办公室具备明显差异化特征，但主应用界面没有与之形成互补关系

用户已经明确确认，希望围绕当前项目做一次**全面、系统、不中断主能力的 UI 改造**，并且要求：

- 改造覆盖主应用的核心组件与子页面，不能只优化局部
- 保持产品专业感，不走廉价“AI SaaS 模板风”
- 保留执行状态沉浸式工作台的独特价值，并让主界面与之形成统一而不冲突的产品体验

## 2. 问题定义

本次变更要解决的并不是单个控件的样式问题，而是以下四类系统性问题：

### 2.1 视觉语言不统一

- 当前主色、字体、背景、阴影、边框、间距体系仍偏默认化
- 聊天页、技能页、登录页的设计语义彼此割裂
- 缺少一套能够在全站复用的视觉 token 和层级规则

### 2.2 主路径缺少高识别度

- 聊天页承担产品核心价值，但欢迎态、消息区、输入区、会话列表的表现仍较通用
- “查看执行状态”已经是功能亮点，但入口层级与主页面叙事仍未形成一致设计
- 技能页和登录页目前更接近功能页，而不是产品形象页

### 2.3 差异化资产没有被放大

- `Star-Office-UI` 像素办公室是极强的体验锚点
- 但主应用如果继续保持标准后台风，会让沉浸式工作台显得像外接页面，而不是产品内能力
- 因此需要建立“主界面克制专业、沉浸层强风格表达”的双层体验结构

### 2.4 后续迭代缺少 UI 规范基础

- 如果本次仅做若干零散样式修改，后续新增功能会继续漂移
- 需要把视觉方向、布局规则、组件边界和性能约束固化为可执行的 change 文档

## 3. 变更目标

### 3.1 产品目标

- 将 `agent-chat` 从“功能完成的工具界面”升级为“有明确产品气质的智能工作台”
- 在不牺牲可用性和信息效率的前提下，显著提升品牌识别度与整体完成度
- 让聊天主路径、技能探索路径、登录入口、执行状态工作台形成统一叙事

### 3.2 设计目标

- 建立一套清晰的全局视觉系统，包括颜色、字体、背景、层级、动效、容器语言
- 将聊天页重构为具有明确主舞台的“工作台式界面”
- 让执行状态入口与沉浸式工作台成为主产品体验中的自然延伸，而不是附属功能
- 将欢迎页、技能页、登录页等次级页面纳入同一视觉语言

### 3.3 工程目标

- 通过主题 token、共享样式与组件级改造降低后续 UI 漂移风险
- 在重做视觉的同时保持当前功能、Mock 方案、真实嵌入方案和测试基础不回退
- 把性能、响应式和可访问性要求前置到设计约束中，而不是事后补救

## 4. 推荐视觉方向

本次变更推荐采用：

> **Paper Ops / 智能工作台** 视觉方向

核心特征：

- 主界面采用克制、专业、带轻微纸面与档案感的工作台风格
- 不走常见的纯蓝白后台，也不走霓虹科技风
- 通过暖白底、深墨文本、靛蓝或铜色强调、轻颗粒纹理、分层纸面容器建立识别度
- 让主聊天界面像一份正在被批注和推进的工作稿
- 让沉浸式像素办公室作为“执行状态的沉浸层”存在，与主界面形成强弱对比

这一路径的核心判断是：

- 主应用负责长期使用效率与专业可信感
- 沉浸式工作台负责差异化表达与情绪价值
- 两者应互补，而不是彼此模仿

## 5. 范围定义

### 5.1 本次纳入范围

- 全局主题 token 与基础视觉变量
- `App` 级 Ant Design token 与主布局骨架
- 左侧侧边栏、用户信息区、会话列表、会话项
- 聊天页主舞台、欢迎页、消息列表、消息气泡、输入区、消息操作
- 图表 / 卡片 / 文件等非纯文本消息容器的视觉统一
- 技能市场页及其筛选、搜索、详情、卡片组件
- 登录页与登录回调承接页的视觉统一
- 执行状态入口、摘要信息层、沉浸式工作台与 iframe-first 视图关系
- 全局动效、空状态、加载态、响应式与可访问性约束

### 5.2 本次不纳入范围

- 重新设计业务流程或修改路由信息架构
- 变更 OpenClaw 会话模型、消息协议或 `Star-Office-UI` 服务端适配协议
- 在本次 change 中增加新的业务模块
- 引入重量级运行时设计系统框架替换当前前端架构

## 6. 方案摘要

### 6.1 全局层

- 建立新的色板、字体栈、背景与阴影规则
- 统一全局页面容器、卡片、面板、标签、状态色和分割线语言
- 清理当前较“默认化”的蓝白灰视觉结果

### 6.2 聊天主路径

- 将聊天页改造成“中央工作台 + 左侧导航脊柱”的结构
- 欢迎态改造成具有任务封面感的工作台首页
- 会话列表从纯功能列表升级为有信息层级的档案区
- 消息区升级为“对话稿批注”语言，图表/卡片消息拥有更清晰的独立容器层
- 输入区与消息舞台形成一体化结构

### 6.3 执行状态体验

- 保持当前“单击直达沉浸式工作台”的交互收敛结果
- 保持 `iframe-first` 的沉浸式办公室展示方式
- 将工作台上的悬浮层保持为极简控制，不再争夺主画面
- 确保主应用与像素办公室在视觉上属于同一产品，而不是拼接感界面

### 6.4 子页面统一

- 技能页升级为与主产品一致的探索式页面，而不是普通卡片列表页
- 登录页升级为具有品牌识别度的入口页
- 空状态、加载态、错误态采用同一视觉语法

## 8. 风险与异议

### 8.1 风险：改造范围大，容易退化为表层换皮

缓解方式：

- 先在 OpenSpec 中固化视觉方向和页面级边界
- 明确覆盖页面与组件清单
- 将“全局 token -> 主舞台 -> 子页面 -> 细节态”作为实施顺序

### 8.2 风险：视觉改造影响现有交互与性能

缓解方式：

- 视觉升级不得破坏既有行为与会话语义
- 对复杂模块维持组件边界，不做无必要重写
- 继续保留已完成的懒加载、保活与 Mock 支撑能力

### 8.3 风险：沉浸式像素办公室与主界面风格冲突

缓解方式：

- 主界面采用克制专业的工作台语言
- 沉浸层保持强风格表达
- 两者通过色板、信息密度和悬浮控件语言建立“同产品、不同场景”的关系

## 9. 成功标准

本次 change 被认为设计完整的标准包括：

- 明确覆盖聊天、技能、登录、沉浸式工作台与全局布局，不遗漏主路径
- 能说明为什么主界面与像素办公室应该采用双层体验而非同风格复刻
- 能明确哪些组件必须统一改造，哪些能力只做视觉承接不改协议
- 能把性能、响应式、可访问性纳入改造约束
- 文档可直接作为后续实施阶段的设计与开发输入
