# 设计文档：agent-chat 产品级 UI 视觉系统升级

## 1. Context

当前仓库已经经历了多轮能力性迭代：

- 聊天主路径已经具备多会话、流式消息、卡片、图表、文件上传等能力
- 执行状态体验已经具备从消息入口直达沉浸式工作台、真实 `Star-Office-UI` 嵌入、同会话 iframe 保活等能力
- 技能市场、登录流程、Mock 链路与测试基础已经存在

但这些能力是在“功能优先”的语境下逐步形成的，因此界面层不可避免地出现了以下问题：

- 全局视觉 token 不稳定，页面更像一组功能模块，而不是一个完整产品
- 页面之间缺少清晰的角色分工和设计叙事
- 局部样式已经多次迭代，但没有形成从全局到组件的统一规则

本次改造需要站在产品层面，而不是单组件层面进行设计。

## 2. Goals / Non-Goals

### 2.1 Goals

- 建立统一的全局视觉系统，并贯穿聊天、技能、登录、执行状态工作台
- 让聊天页成为明确的“主工作台”，而不是普通消息列表页
- 让 `Star-Office-UI` 作为沉浸式能力层，与主界面形成自然互补
- 让视觉升级覆盖主要组件与子页面，避免出现“主页面高级、次级页面掉队”的结果
- 保持现有功能、协议和状态模型稳定
- 将性能、响应式、可访问性作为设计输入的一部分

### 2.2 Non-Goals

- 不在本次变更中重构消息协议、会话模型或服务端适配协议
- 不在本次变更中替换 Ant Design 或引入新的重型 UI 框架
- 不在本次变更中新增业务模块或修改信息架构
- 不在本次变更中重做 `Star-Office-UI` 页面本体

## 3. 核心设计决策

### Decision 1：采用“Paper Ops / 智能工作台”作为唯一视觉主轴

本次不采用常见“蓝白科技风”、不采用过度娱乐化像素风、也不采用极度冷淡的纯极简样式。原因如下：

- 蓝白科技风过于通用，无法突出产品个性
- 全站像素化会直接损害长期办公场景下的可读性与效率
- 纯极简会削弱本产品已经具备的执行状态故事感

因此采用中间路线：

- 主界面：克制、专业、偏编辑式工作台
- 沉浸层：保留真实像素办公室的戏剧性

### Decision 2：先重构“主舞台”，再统一子页面

主舞台包括：

- 全局布局
- 左侧导航与会话区
- 聊天主内容区
- 输入区与消息区
- 执行状态入口

原因：

- 用户最常停留在这里
- 主舞台定调后，技能页与登录页更容易继承相同语言
- 如果先做子页面，容易再次产生风格漂移

### Decision 3：将执行状态视为“产品内沉浸层”，而不是普通附件面板

这意味着：

- 聊天页中的执行状态入口应轻量且明确
- 真正承载价值的是沉浸式工作台
- 主页面不再试图和真实像素办公室争夺主视觉

这与之前已经确定的交互方案一致：

- 查看执行状态单击直达工作台
- 工作台 `iframe-first`
- 关闭时隐藏保活，不重复初始化

### Decision 4：视觉升级必须以 token 和组件边界收敛，而不是写死页面样式

需要优先收敛：

- 颜色 token
- 字体 token
- 背景 / 边框 / 阴影 token
- 间距和圆角 token
- 状态色和反馈色

否则后续改造会继续在组件局部发散。

## 4. 页面与模块级方案

### 4.1 全局 App / Layout

#### 目标

- 统一主应用壳层、背景、主色、字体与容器语言
- 让 `AppShell`、`MainLayout`、Ant Design token 与样式变量指向同一设计系统

#### 方案

- 重置 `src/styles/variables.less`，建立新的视觉 token 命名与默认值
- 调整 `src/App.tsx` 中 Ant Design `ConfigProvider` token
- 在 `src/styles/global.less` 中补充页面背景、字体、滚动条与选择态等全局规则
- 对 `MainLayout` 与 `Sidebar` 做结构性视觉收敛，使左侧区域更像“导航脊柱”

### 4.2 聊天页

#### 目标

- 把聊天页提升为产品主舞台
- 同时承载效率、阅读性和品牌感

#### 方案

- 页面背景引入轻纹理与层次，而不是平涂底色
- 中央聊天列做成“工作稿画布”，宽度更聚焦
- 顶部补充会话标题、当前上下文和轻量状态锚点
- 输入区与消息区在视觉上形成整体，而不是两个孤立层

### 4.3 欢迎页

#### 目标

- 把欢迎态从“快捷按钮墙”升级为“今日工作台封面”

#### 方案

- 主标题采用更强识别度的展示字体
- 快捷建议卡片强调任务类型、入口语义与场景感
- 卡片布局可加入受控错位、编号标签和边角细节，形成桌面文档感

### 4.4 会话列表与会话项

#### 目标

- 把历史会话从普通列表升级为带摘要层级的档案区

#### 方案

- 强化当前会话与普通会话的层次差异
- 增加时间、摘要、状态信息的排布优先级
- 新建对话按钮升级为高辨识度主操作
- 空状态与滚动区风格统一到新的侧边栏语义

### 4.5 消息气泡与非文本内容

#### 目标

- 建立“用户提交 / 助手批注 / 图表结果 / 卡片动作 / 文件上传”的多层消息语言

#### 方案

- 用户消息使用更稳重的强调色，不再是默认高饱和蓝
- 助手消息采用纸面批注式气泡，降低默认白卡感
- 图表和卡片消息引入更清晰的结果容器层
- 消息操作、时间戳、状态反馈应贴近气泡边缘，避免孤立悬浮

### 4.6 技能页

#### 目标

- 让技能市场从“功能列表页”升级为产品探索页面

#### 方案

- 将筛选、搜索、卡片、详情统一到新的视觉语言
- 强化技能卡的封面层、能力标签和动作层次
- 让技能页与聊天主界面在节奏上形成一致，而不是完全割裂

### 4.7 登录页

#### 目标

- 登录页成为品牌入口页，而不是单纯承接 OAuth 按钮

#### 方案

- 采用更强的封面式布局
- 保持信息简洁，但建立明确的品牌基调
- 回调页与登录页保持一致的加载和状态语言

### 4.8 执行状态工作台

#### 目标

- 保持已确认的交互方案，同时让主产品与沉浸式工作台形成一致感

#### 方案

- 继续坚持 `iframe-first`
- 悬浮控件极简化，只保留返回、收起、必要状态
- 将工作台外层背景、过渡动画、按钮语义与主产品色板对齐
- 不在工作台中重复堆叠主产品界面元素

## 5. 设计 Token 建议

以下为方向性约束，实施时可在具体文件中细化：

### 5.1 色板

- 主背景：暖白、纸白、浅沙色
- 内容文本：深墨灰，不用纯黑
- 强调色：靛蓝或铜色二选一作为主品牌强调
- 辅助色：鼠尾草绿、琥珀、砖红用于状态反馈

### 5.2 字体

- 标题字体：具备性格的中文展示字体
- 正文字体：高可读中文 sans-serif
- 等宽字体：保留代码与技术内容可读性

### 5.3 容器与边界

- 少用“默认 SaaS 大圆角 + 大投影”
- 通过纸面层次、边框、轻阴影、局部纹理和标签感建立区分
- 各类容器要有统一的边界语言

### 5.4 动效

- 页面级进入动画必须有，但要克制
- 优先投入在欢迎页进入、消息出现、状态切换、工作台切换
- 不做无意义的全局持续动效

## 6. 性能与可访问性约束

### 6.1 性能

- 视觉增强不能破坏现有懒加载策略
- 避免无意义的大图背景与持续动画
- 对 `Visualize`、图表、技能详情等重模块保持按需加载意识

### 6.2 响应式

- 桌面宽屏为第一优先级
- 中等宽度视口下需要保证聊天主舞台不崩塌
- 登录页、技能页、欢迎页不得只在大屏下成立

### 6.3 可访问性

- 颜色对比度必须可读
- 交互焦点态需要可见
- 关键按钮、导航、输入区需要保持明确可点击反馈

## 7. 风险 / Trade-offs

- 范围较大，容易引入样式回归
  - 通过分阶段任务、组件级测试和视觉边界文档控制

- 视觉表达增强可能与 Ant Design 默认样式冲突
  - 通过 token、局部覆写和共享 Less 变量逐步接管，而不是全面对抗框架

- 聊天主舞台与沉浸式工作台可能再次出现风格断层
  - 通过色板、悬浮按钮和过渡动效建立连续性

## 8. Migration Plan

1. 创建并评审本 OpenSpec change
2. 先改全局 token、布局骨架和聊天主舞台
3. 再改欢迎页、会话列表、消息体系和输入区
4. 再覆盖技能页、登录页和沉浸式工作台外层体验
5. 最后统一空状态、加载态、动效与测试
6. 更新阶段六及相关总文档

## 9. Open Questions

当前无需新增必须由用户回答的阻塞性问题，原因是：

- 视觉方向已经明确确认
- 执行状态交互方案已经在前序 change 中确认
- 本次 change 的主要不确定性属于实施细节，而不是方案边界

后续如需二次确认，可在实施阶段就以下问题进行收敛：

- 具体主强调色采用靛蓝还是铜色
- 展示字体最终采用哪套可部署字体方案
