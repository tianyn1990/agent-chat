# 变更提案：引入真实 Star-Office-UI 侧车集成方案

> Change ID: `add-real-star-office-sidecar`
> 状态：提案中
> 创建日期：2026-03-24

## 1. 背景

阶段六当前已经具备两类相关前置背景：

- 前端已经具备可视化入口、右侧面板与 `/visualize/:sessionId` 页面
- 仓库中已经并行提出 `add-local-star-office-mock` change，用于提供同域 mock 验证能力

这为“iframe 承接是否可行”“多会话绑定是否串线”的验证提供了基础，但还没有解决一个更直接的产品问题：

- 用户希望在本地和联调环境中看到 **真实 Star-Office-UI 的像素风办公室效果**
- 而不是只看到一个用于验证协议和布局的开发期 mock 页面

因此，当前阶段需要补充一个新的独立 change，专门回答以下问题：

1. 如何在当前工程体系中接入 **真实上游 Star-Office-UI**
2. 如何让它以 **独立服务** 的方式被 `agent-chat` iframe 承接
3. 如何在 **多会话** 场景下仍然保持会话级语义正确
4. 如何在 **本地开发** 时看到真实像素办公室效果
5. 如何规避 **许可证**、升级、部署和维护风险

## 2. 问题定义

当前本地 mock change 的定位是：

- 验证执行状态入口
- 验证会话隔离
- 验证 iframe 承接
- 验证未来真实适配层的契约边界

它的定位从来不是：

- 复刻 Star-Office-UI 的视觉表现
- 替代真实像素办公室
- 作为最终交付形态长期存在

现在的问题变成了另一个层面：

- 真实 Star-Office-UI 是一个 **独立项目**
- 它并不是 React 组件库，而是完整的 `Flask + 静态前端 + Phaser` 应用
- 它原生只理解 `/status`、`/agents` 这类状态接口，并不理解 `sessionId`
- 它的美术资源存在额外许可证边界，不能直接草率地复制进主应用代码中

所以，本次 change 的目标不是“继续增强本地 mock”，而是要定义一套 **真实上游集成方案**。

## 3. 变更目标

### 3.1 产品目标

- 用户在本地开发与目标环境中都可以看到真实 Star-Office-UI 的像素办公室效果
- `agent-chat` 保持“查看执行状态”的会话级语义
- 用户无需离开当前页面，通过 iframe 即可获得沉浸式查看体验

### 3.2 架构目标

- 保持 `agent-chat` 与 Star-Office-UI 的 **服务解耦**
- 通过 **服务端适配层** 解决会话语义，而不是让前端硬塞 `postMessage`
- 本地开发、测试环境、生产环境尽量复用同一套拓扑
- 对上游升级保持清晰边界，不把整个外部项目无差别揉进主前端包

### 3.3 风险控制目标

- 明确区分“主项目代码”“上游依赖代码”“适配层代码”
- 明确记录许可证风险，特别是像素美术资源的商用限制
- 为未来替换素材、自研外观或切换供应方式预留空间

## 4. 方案范围

### 4.1 本次纳入范围

- 真实 Star-Office-UI 的引入策略与仓库管理方式
- 本地开发、测试、生产三类环境的服务拓扑
- 同域 iframe 嵌入方案
- `sessionId` 到 Star-Office 状态接口之间的适配责任划分
- 许可证、运维、升级和回滚策略
- 与阶段六主文档、服务端适配层文档、部署指南之间的关系梳理

### 4.2 本次不纳入范围

- 立即实现真实 Star-Office-UI 的代码接入
- 立即编写完整容器脚本或 CI/CD 流程
- 立即改造上游所有前端资源路径
- 立即实现 OpenClaw 到 Star-Office-UI 的完整状态同步逻辑
- 立即替换上游商业受限素材

## 5. 核心判断

### 5.1 不建议把 Star-Office-UI 重写成 React 组件

原因：

- 上游不是组件库，而是完整应用
- 像素角色动画与场景逻辑依赖 Phaser 运行时
- 重写成本高，且会丢失与上游同步升级能力
- 一旦重写，本地“真实效果”目标反而失效

### 5.2 不建议直接把上游源码和素材复制进主应用目录

原因：

- 升级路径差，后续难以追踪上游变更
- 容易让主仓库承担不必要的许可证风险
- 责任边界模糊，不利于后续替换素材或改造部署

### 5.3 推荐采用“真实上游侧车服务 + 会话适配层 + 同域 iframe”

原因：

- 这是唯一能保留真实像素办公室视觉效果的路线
- 也是最符合上游项目原始运行方式的路线
- 可以同时满足本地开发、测试联调和生产嵌入

## 6. 推荐方案摘要

推荐将本次 change 定义为：

> 在现有本地 mock change 之外，新增一条“真实 Star-Office-UI 侧车接入”路线。

核心做法：

1. 以 **独立 sidecar 服务** 运行真实 Star-Office-UI
2. 在工程层面优先采用 **固定 commit 的子仓库或受控 fork** 管理上游来源
3. 由 **服务端适配层** 负责将 `agent-chat sessionId` 翻译成 Star-Office 所需状态接口
4. 由 `agent-chat` 前端通过 **同域 iframe** 承接显示
5. 本地开发时通过 **真实上游服务 + 本地适配层** 提供真实像素办公室效果

## 7. 为什么要作为独立 change

这件事与 `add-local-star-office-mock` 的目标不同：

- `add-local-star-office-mock` 解决的是“无服务时如何本地验证”
- `add-real-star-office-sidecar` 解决的是“如何接入真实上游并看到真实像素效果”

两者应该同时存在，而不是互相替代：

- 前者是开发兜底与测试夹具
- 后者是真实接入路线与最终落地方向

把它们拆成两个 change，有三个好处：

1. 决策边界清楚，不会把 mock 和真实接入混为一谈
2. 可以分别评审许可证、部署和工程成本
3. 后续实现时可以分阶段推进，不会把风险集中到一次提交

## 8. 风险与异议

### 8.1 许可证风险

需要重点记录：

- 上游代码逻辑可以按较宽松方式复用
- 但像素美术资源存在非商业使用限制

这意味着：

- 如果当前项目或未来部署存在商业属性，不能默认把这些资源直接打进正式交付物
- 必须在实现前完成法务确认、资产替换方案或上游授权确认

### 8.2 上游可升级性风险

如果采用粗暴复制方式：

- 每次升级都要手工 diff
- 本地 patch 容易漂移
- 回滚困难

因此需要明确选择受控 fork、submodule 或 subtree 之一，并固定升级流程。

### 8.3 会话语义风险

Star-Office-UI 原生没有 session 概念。

如果没有服务端适配层：

- iframe 中只能看到全局状态
- 不能正确回答“当前展示的是哪个会话”
- 多会话就会退化为伪多会话

## 9. 成功标准

本次 change 评审通过的标准：

- 能清晰回答“为什么本地 mock 还不够”
- 能清晰回答“如何看到真实像素办公室效果”
- 能清晰回答“为什么推荐 sidecar，而不是 React 重写或源码硬拷贝”
- 能清晰回答“多会话下 iframe 到底展示哪个会话”
- 能清晰回答“许可证边界在哪里”
- 文档可作为后续实现的直接输入
