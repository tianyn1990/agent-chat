# 设计文档：执行状态一键直达与 iframe-first 沉浸式工作台

## Context

在上一轮变更中，系统已经完成了三件关键事情：

- 将真实 `Star-Office-UI` 与本地高保真联调能力接入前端
- 建立沉浸式工作台与同会话 `iframe` 保活机制
- 将聊天页右侧从“直接承载真实像素办公室”改造为“摘要侧栏 + 工作台入口”

这解决了“右侧侧栏无法完整承载真实办公室页面”和“关闭后再次打开要重复初始化”的工程问题，但从用户体验看，仍然存在结构性偏差：

1. 主功能路径过长  
   用户真正想看的内容是沉浸式像素办公室，但现在必须先经过摘要侧栏，再进入工作台。

2. 辅助信息权重过高  
   摘要信息只是辅助，不应以大面积右侧侧栏形态长期占据聊天主空间。

3. 工作台仍未把 `iframe` 置于第一优先级  
   当前工作台有较重的标题、说明、提示等壳层内容，真实 `iframe` 没有拿到足够多的视口。

4. 退出路径不应依赖浏览器  
   用户建议提供悬浮在 `iframe` 上层的返回控件，这是合理的。工作台是应用内部体验的一部分，退出应当由应用自身明确提供。

## Goals / Non-Goals

- Goals:
  - 将“查看执行状态”调整为一键直达沉浸式工作台
  - 将摘要视图降级为轻量提示，而不是大面积侧栏
  - 让沉浸式工作台成为真正的 `iframe-first` 界面
  - 在 `iframe` 上层提供应用内返回、收起、刷新等悬浮控件
  - 保留现有同会话 `iframe` 保活机制，不回退已有性能收益
  - 保持工作台仍可与 `/visualize/:sessionId` 路由同步

- Non-Goals:
  - 不在本次变更中实现服务端适配层
  - 不在本次变更中实现多会话多工作台并行缓存池
  - 不在本次变更中实现复杂缩放、全景拖拽或画中画模式
  - 不在本次变更中移除现有工作台路由

## Decisions

### Decision: 主入口改为“一键直达沉浸式工作台”

新的主交互为：

- 用户在消息操作区点击“查看执行状态”
- 系统直接打开当前会话的沉浸式工作台
- 摘要不再作为主路径的中间步骤

原因：

- 执行状态工作台才是用户关注的核心页面
- 两级跳转会明显拉长感知路径
- 已有 `iframe` 保活能力后，直接打开工作台的成本可控

### Decision: 摘要侧栏降级为轻量状态提示

摘要信息仍然保留，但不再采用现有大面积右侧侧栏形态。

推荐落地方向：

- 会话头部状态胶囊
- 聊天区右上角轻量悬浮提示
- 或只在工作台关闭后保留一个可恢复入口

它只承担三类职责：

- 展示当前会话的简短状态
- 提供重新展开工作台的入口
- 在工作台隐藏但仍保活时，提示用户当前工作台仍可恢复

它不再承担：

- 独立承接完整说明内容
- 作为进入工作台的主流程容器
- 长时间占据聊天主布局

### Decision: 沉浸式工作台采用 iframe-first 布局

工作台应尽可能把视口让给真实 `Star-Office-UI` 页面：

- `iframe` 占满主要区域
- 非必要说明、标题、提示不再占据独立大块纵向空间
- 应用壳层收敛为悬浮覆盖层，而不是挤占内容流布局

建议布局：

- 全屏固定层承载工作台
- `iframe` 铺满整个工作台容器
- 顶部左上角悬浮工具条显示：
  - 返回聊天
  - 收起
  - 刷新
  - 会话标识
  - 状态标签

可选信息面板应默认折叠，不默认展开占据面积。

### Decision: 返回能力由应用内悬浮控件提供

本次变更明确不把浏览器原生返回当作主要退出方式。

工作台至少提供以下应用内控件：

- 返回聊天
- 收起工作台
- 刷新工作台

其中：

- “返回聊天”负责关闭工作台可见态并回到当前会话聊天页
- “收起工作台”负责隐藏工作台但保留上下文
- 浏览器返回仍可作为兼容性路径存在，但不是主交互

这样更符合用户心智，也能减少对浏览器历史栈的依赖。

### Decision: 保活策略继续沿用“最近工作台实例”

本次变更不改变现有保活边界：

- 继续保留当前最近一次打开的会话工作台实例
- 关闭时只隐藏，不销毁
- 同会话再次进入时直接复用

原因：

- 本次目标是修正交互，而不是扩展缓存池复杂度
- 现有保活策略已经能解决“重复初始化”这一关键问题

## Risks / Trade-offs

- 风险：摘要被弱化后，用户可能缺少“侧边持续观察状态”的感觉  
  - 缓解：保留轻量状态胶囊或悬浮提示，并在工作台关闭后提供明显恢复入口

- 风险：悬浮工具条可能遮挡 `iframe` 顶部内容  
  - 缓解：工具条控制尺寸、透明度和安全边距，只占用很小区域，并允许自动收起或淡化

- 风险：直接打开工作台会让聊天页停留时间减少  
  - 缓解：关闭工作台时保持会话上下文不丢失，并提供快速返回聊天的应用内按钮

- 风险：路由与覆盖层并存时状态同步更复杂  
  - 缓解：继续由全局 store 统一维护 `sessionId`、工作台可见态和保活实例

## Migration Plan

1. 创建本次 OpenSpec change 并获得确认
2. 将“查看执行状态”主路径改为直接打开工作台
3. 将现有摘要侧栏改造成轻量状态提示
4. 重构工作台布局，改为 `iframe-first + overlay controls`
5. 将返回/收起/刷新控件收敛到 `iframe` 上层悬浮层
6. 保持现有 `iframe` 保活逻辑与路由同步逻辑
7. 补充测试并更新阶段六文档

## Alternatives Considered

### 方案 A：继续保留摘要侧栏作为主入口

不采纳。

原因：

- 仍然是两级跳转
- 侧栏仍然占用过大面积
- 用户已经明确认为这条路径繁琐

### 方案 B：完全移除摘要能力，只保留工作台

暂不采纳。

原因：

- 轻量状态提示仍有价值
- 工作台关闭后需要一个低成本恢复入口
- 某些场景下用户只想知道“当前是否在运行”，不一定立刻进入工作台

### 方案 C：依赖浏览器返回退出工作台

不采纳为主交互。

原因：

- 浏览器返回不属于产品内显式能力
- 在 `iframe`、深链接和会话切换场景下心智不稳定
- 用户已经明确提出希望使用应用内悬浮按钮

## Open Questions

- 轻量摘要提示最终采用“会话头部胶囊”还是“聊天区悬浮小卡片”
- 悬浮工具条是否需要自动隐藏/悬停展开
- 未来是否要支持多会话工作台缓存池
