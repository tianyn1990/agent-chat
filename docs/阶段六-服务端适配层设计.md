# 阶段六：服务端适配层设计

> 目标：把 `agent-chat` 的会话语义、OpenClaw 的 session 模型、Star-Office-UI 的状态看板模型接在一起，形成真实可用的会话级执行状态能力。

## 1. 背景

当前系统有三个不同层次的“会话/状态”概念：

### 1.1 agent-chat 前端

- 以 `sessionId` 组织会话列表、消息列表、草稿与发送状态
- 用户点击消息上的“查看执行状态”时，天然期待看到该会话对应的执行过程

### 1.2 OpenClaw Gateway

- 以 gateway session 为真实上下文边界
- 同一 session 内串行，不同 session 可并行
- gateway 是 session 的唯一事实来源

### 1.3 Star-Office-UI

- 以 `/status` 展示主 Agent 状态
- 以 `/agents` 展示多 Agent 协作状态
- 原生只有状态机，没有 session 概念

问题就在这里：

- `agent-chat` 有 session
- OpenClaw 有 session
- Star-Office-UI 没有 session

所以必须增加一个适配层，负责翻译。

## 2. 适配层职责

适配层的最小职责有四项：

1. 建立 `agent-chat sessionId -> OpenClaw sessionKey/sessionId` 映射
2. 监听 OpenClaw 事件，理解当前 session 正在做什么
3. 将 OpenClaw 事件压缩为 Star-Office-UI 能接受的状态机
4. 对外提供“按 session 查看执行状态”的查询能力

## 3. 为什么不能直接用前端 iframe 解决

不能靠 iframe 前端层解决，原因如下：

- Star-Office-UI 不是嵌入式组件，而是完整独立服务
- 它依赖服务端接口 `/status` `/agents`
- `postMessage` 只能传值，不能让它天然理解 OpenClaw session
- 真正的会话隔离与并发信息只存在于 gateway

结论：

- 前端 iframe 只负责承接
- 语义正确性必须由服务端保证

## 4. 推荐总体架构

```text
agent-chat browser
  └─ 右侧面板 /visualize/:sessionId
       └─ iframe -> Star-Office-UI

agent-chat backend / gateway adapter
  ├─ 读取或创建前端 sessionId
  ├─ 维护到 OpenClaw session 的映射
  ├─ 监听 gateway 事件
  └─ 向 Star-Office-UI 推送状态

OpenClaw Gateway
  └─ 作为真实 session 与执行事件来源

Star-Office-UI
  └─ 渲染状态看板
```

## 5. 数据模型

### 5.1 会话映射表

建议维护一张映射表：

```ts
interface SessionBinding {
  chatSessionId: string;
  openclawSessionKey: string;
  openclawSessionId?: string;
  createdAt: number;
  updatedAt: number;
}
```

如果网关本身已有可查询的 session list，优先用网关数据做主表；如果没有，则由业务后端维护。

### 5.2 可视化状态表

```ts
type VisualizeState =
  | 'idle'
  | 'researching'
  | 'writing'
  | 'executing'
  | 'syncing'
  | 'error';

interface SessionVisualizeState {
  chatSessionId: string;
  state: VisualizeState;
  detail: string;
  updatedAt: number;
  source: 'gateway' | 'frontend-fallback';
  messageId?: string;
}
```

这张表的用途是：

- 给 `agent-chat` 面板提供会话级文案
- 给 Star-Office-UI 提供当前状态推送

## 6. 状态映射规则

建议先采用有限状态机，而不是试图一步到位做 DAG。

### 6.1 推荐映射

| OpenClaw / 业务事件 | Star-Office-UI 状态 | detail 示例 |
|---|---|---|
| 用户消息入队 | `researching` | 正在分析用户请求 |
| LLM 正在输出 | `writing` | 正在生成回复 |
| 执行工具/命令 | `executing` | 正在执行命令 |
| 上传/同步/推送 | `syncing` | 正在同步外部状态 |
| 正常完成 | `idle` | 本轮任务已完成 |
| 异常中断 | `error` | 任务执行失败 |

### 6.2 detail 生成规则

detail 需要兼顾可读性与隐私：

- 不直接透传敏感原文
- 最长建议 80~120 个字符
- 对路径、token、业务隐私做脱敏

## 7. 接口设计

### 7.1 agent-chat / 适配层内部接口

建议提供如下接口：

```http
GET /api/visualize/sessions/:chatSessionId
```

返回：

```json
{
  "chatSessionId": "session_123",
  "openclawSessionKey": "agent:main:webchat:channel:abc",
  "runtime": {
    "state": "writing",
    "detail": "正在生成回复",
    "updatedAt": 1742745600000
  }
}
```

### 7.2 Star-Office-UI 推送接口

现阶段优先复用它已有的接口：

- 主 Agent 状态：`POST /set_state`
- 多 Agent 协作：`POST /agent-push`

推荐策略：

- 单会话查看场景：由业务层维护“当前查看 session”的状态并推送主 Agent 状态
- 多会话/多 Agent 场景：将不同会话映射为不同访客 Agent，推送到 `/agent-push`

## 8. 两种落地策略

### 8.1 策略 A：单会话聚焦模式

特点：

- 页面一次只服务一个会话
- 当前打开的 `sessionId` 对应主 Agent 状态
- 其他会话不在 Star-Office-UI 中展示

优点：

- 实现最简单
- 和当前“右侧面板查看本会话”语义一致

缺点：

- 不适合同时观察多个活跃会话

### 8.2 策略 B：多会话映射为多 Agent

特点：

- 每个活跃会话在 Star-Office-UI 中变成一个访客 Agent
- 主 Agent 可代表当前焦点会话

优点：

- 更接近“多会话并行”的真实状态
- 能直观看出不同会话的忙闲分布

缺点：

- 容易把“会话”和“Agent”混淆
- 需要管理 joinKey、agentId、清理策略

### 8.3 推荐

阶段六推荐先做 **策略 A**。

理由：

- 用户已经在某个会话内点开面板
- 需求重点是“当前会话是否在执行、执行到哪一步”
- 先保证语义正确，再谈多会话并行可视化

## 9. 生命周期设计

### 9.1 打开面板

1. 前端传入 `chatSessionId`
2. 适配层查询绑定关系
3. 返回当前 runtime
4. 前端渲染会话上下文与 iframe

### 9.2 执行中

1. OpenClaw 事件流更新 runtime
2. 适配层持久化会话状态
3. 同步推送到 Star-Office-UI
4. 前端可轮询或订阅自己的 runtime 接口

### 9.3 完成/异常

1. 会话状态回到 `idle` 或 `error`
2. 超时后自动回 `idle`
3. 面板仍可保留最近一次 detail，用于复盘

## 10. 容错与降级

### 10.1 Star-Office-UI 不可用

前端显示：

- 会话 ID
- 当前本地 runtime
- 未配置或不可访问提示

### 10.2 适配层未上线

前端仅展示本地 fallback 状态：

- 用户发消息后标 `researching`
- 收到流式块时标 `writing`
- 完成后标 `idle`
- 错误时标 `error`

这就是当前阶段已经接入的前端骨架语义。

### 10.3 OpenClaw 无多 session 映射

必须显式标记：

- 当前系统仅支持“单主会话上下文”
- 不允许继续对外宣称“会话级执行状态”

## 11. 安全与隐私

- 不在 detail 中暴露敏感命令参数、认证信息、业务数据
- 避免将用户原始问题全文直接推送到 Star-Office-UI
- session 映射表需带 TTL 或归档机制
- 适配层只暴露必要字段，不泄露 OpenClaw 内部路径与 host 信息

## 12. 验收标准

- [ ] `chatSessionId` 与 OpenClaw session 绑定稳定
- [ ] 打开某会话面板时，看到的是该会话状态而不是全局随机状态
- [ ] OpenClaw 执行中，状态能正确切换为 `researching/writing/executing`
- [ ] 任务完成后回到 `idle`
- [ ] 错误时能进入 `error`
- [ ] Star-Office-UI 不可用时，前端仍有可用降级
