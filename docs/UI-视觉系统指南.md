# agent-chat UI 视觉系统指南

## 1. 文档目的

本文档用于沉淀 `agent-chat` 当前阶段已经确认并落地的 UI 视觉系统，作为后续新增页面、组件与功能模块时的统一参考。

目标不是限制实现自由度，而是避免以下问题反复出现：

- 新页面重新回到“默认后台风”
- 新组件沿用另一套颜色、圆角、阴影和信息层级
- 主聊天工作台、技能页、登录页、执行状态工作台之间再次出现风格断层
- 只做局部好看，整体气质重新发散

## 2. 视觉主轴

当前项目采用的唯一主视觉方向为：

> `Paper Ops / 智能工作台`

上一阶段的收紧版落点为：

> `Paper Ops / Flat Console`

当前阶段继续推进后的落点为：

> `Paper Ops / Graphite Console`

在引入双主题系统后，主应用当前采用的正式视觉组合为：

- `dark`：`Paper Ops / Graphite Console`
- `light`：`Ivory Editorial / Paper Console`

核心判断：

- 主应用不是科技霓虹风，也不是通用蓝白后台风
- 主界面强调“纸面工作台、档案感、批注感、编辑式推进”
- 在此基础上继续收紧为更扁平、更紧凑、更偏操作台的界面
- 当前主应用进一步转向深 graphite 壳层、更薄 chrome 和更强主舞台
- 沉浸式 `Star-Office-UI` 像素办公室属于产品的“沉浸层”，不是主应用的直接复制对象

因此需要始终保持：

- 主界面：克制、专业、可长时间使用
- 主界面 chrome：更薄、更少阴影、更少厚卡片感
- 主框架：优先采用单一工作台底板与线性分隔，而不是左右双大壳体
- 主舞台优先于侧栏与说明区，欢迎态和对话态都应围绕中心画布与底部 dock 组织
- 沉浸层：保留更强的风格表达和戏剧性

## 2.1 双主题原则

双主题不是“两套互不相关的皮肤”，而是一套共享布局秩序的产品视觉系统：

- `dark` 负责深 graphite 工作台语义，适合长时间沉浸操作
- `light` 负责暖白纸面工作台语义，强调编辑、文档与知识整理感
- 两种主题必须共享相同的信息架构、组件语义和层级规则
- 允许更换背景、文本、表面层、CTA 和图表呈现方式
- 不允许因为切到 `light` 就回退成默认蓝白后台风
- 不允许因为接入双主题而重新破坏当前 `dark` 已落地的结构和观感

## 3. 页面层级原则

### 3.1 主工作台优先

聊天页是产品主舞台，设计优先级最高：

- 会话标题、上下文说明、消息画布、输入区要形成一个完整工作台
- 不要把聊天页做成普通“左侧栏 + 中间列表 + 下方输入框”的通用套壳

### 3.2 子页面必须继承主叙事

技能页、登录页、执行状态工作台都必须属于同一产品体系：

- 技能页是“能力扩展页”，不是普通卡片列表
- 登录页是“品牌入口页”，不是只放一个登录按钮
- 工作台是“沉浸层”，不是把主应用再复制一遍

### 3.3 工作台与沉浸层分工

执行状态相关体验遵循以下边界：

- 聊天页中的执行状态入口应轻量明确
- 沉浸式工作台坚持 `iframe-first`
- 工作台外层控件极简，不得反向挤占真实像素办公室画面

## 4. 设计 Token 约束

### 4.1 色彩

主色：

- 主品牌色使用偏稳重的靛蓝系
- 局部强调使用铜棕色系
- 当前主背景以 graphite / ink 深色壳层为主，Paper Ops 的暖感只保留在局部强调与品牌记忆中
- `light` 主题下背景允许转为暖白 / parchment / ivory，但仍需保持低饱和靛蓝与铜棕作为主辅记忆点

禁止：

- 回到标准高饱和蓝白后台配色
- 大面积使用没有上下文的紫色渐变
- 为了“科技感”堆叠高亮霓虹色

### 4.2 字体

当前约束：

- 标题优先使用更具识别度的 display serif 风格
- 正文保持高可读 sans-serif
- 代码与技术内容使用等宽字体

新增模块时：

- 不要把所有标题都退回系统 sans-serif 粗体
- 不要为局部页面单独引入另一套风格完全不同的字体语言

### 4.3 容器语言

统一采用：

- 深 graphite 主画布与低对比表面层
- 轻边框
- 克制阴影
- 中小圆角
- 框架层优先通过分隔线、留白和局部弱表面层建立结构
- 通过明度差、分隔和对齐建立层级，而不是依赖大阴影

在 `light` 主题下对应为：

- 暖白纸面主画布与更浅一级表面层
- 细边框与浅分隔线
- 极弱阴影，优先通过底色变化和线性分隔表达结构
- 仍然避免大面积厚卡片

禁止：

- 大面积纯白卡片 + 灰边的默认 SaaS 卡片感
- 过厚的玻璃态、漂浮感和大面积 glow
- 所有区域都使用相同容器导致信息层级消失

### 4.4 密度规则

当前主应用默认应遵循“中高信息密度”的工作台原则：

- 页面头部优先压缩装饰性留白，把空间让给正文与操作
- 统计块、提示块、建议卡优先做成紧凑摘要，而不是大卡片
- 标题需要有识别度，但不能压过内容本身
- 正文、表单、列表的点击热区可以稳定，但不应依赖过大 padding 制造高级感

## 5. 组件设计约束

### 5.1 布局组件

适用于：

- `Layout`
- 页面头部
- 区域面板

要求：

- 通过留白、信息密度和低对比表面层建立层级
- 主次区域必须明显，不要平均分配视觉注意力
- 页面不应出现多个同级“厚重盒子”并列争抢焦点
- 导航应优先采用 rail-first 思路，让主舞台成为页面第一主角
- 主布局优先采用“框架级去壳、内容级保台”的策略：删除整页大壳体，保留消息画布、结果区和工具面等真正承担认知分组的主舞台

### 5.2 聊天消息组件

要求：

- 用户消息与助手消息必须有明确角色差异
- 图表、卡片、文件结果必须拥有独立结果容器语义
- 时间戳、消息操作、状态反馈应贴边收纳，不要成为漂浮附件

禁止：

- 把所有消息都做成一模一样的白色方卡
- 图表和卡片退化成普通文本块

### 5.3 输入组件

要求：

- 输入区要像“批注条”或“工作台操作条”
- 当前阶段默认将输入区视为底部 dock，而不是独立厚卡片
- 文件上传预览与文本输入必须属于同一表面层
- 附件按钮与附件预览默认应与输入框保持同一条 dock，不要把附件单独拆成第二行说明区
- 焦点态要清晰可感知
- 禁止把输入区继续做成厚重漂浮卡片

### 5.4 技能组件

要求：

- 技能卡要体现“能力模块”而不是普通商品卡
- 搜索、筛选、详情抽屉要和主视觉系统一致

### 5.5 工作台控件

要求：

- 沉浸式工作台悬浮控件必须极简
- 允许有返回、收起、刷新
- 优先沿外层边缘布置，保证真实 iframe 的可视面积
- 主操作要清晰可发现，次操作要收纳
- 不允许重新铺大面积说明文字或大头部占位

### 5.6 侧栏与列表密度

要求：

- 会话列表、技能列表、档案列表必须有稳定且明确的纵向间距，避免条目彼此粘连
- 列表项优先通过间距、轻边框和背景明度差建立层级，不依赖重阴影
- 侧栏滚动容器需要避免横向溢出，并为滚动条预留最小安全间距
- 侧栏展开的二级 panel 应表现为附着在 rail 旁的工作台分区，而不是嵌套在大卡片里的第二块卡片

### 5.7 按钮与 CTA

要求：

- 主 CTA 优先使用低饱和靛蓝填充或半填充方案，而不是高亮渐变蓝
- 次级按钮优先保持深色表面层与轻边框，不要出现突兀的白底高亮按钮
- `light` 主题下次级按钮也应保持 paper / stone 表面层，不允许出现高饱和默认 AntD 蓝按钮
- `danger` 操作也应遵循低饱和警示色方案，避免鲜艳红色破坏整体克制感
- 同一路径中的按钮视觉强度必须一致，例如聊天发送、技能安装、登录入口、新建对话不能各自使用不同色温和饱和度

## 6. 动效与反馈

允许：

- 页面进入时的轻量过渡
- 卡片 hover 与消息出现的细微位移
- 输入焦点、按钮 hover、状态切换的清晰反馈

禁止：

- 长时间持续动画
- 与业务无关的炫技式动效
- 大范围闪烁、强缩放、无意义浮动

## 7. 响应式与可访问性

### 7.1 响应式

优先级：

1. 桌面宽屏
2. 中等宽度桌面
3. 再向下收敛移动端

要求：

- 聊天主舞台不能在中等宽度下崩塌
- 技能页、欢迎页、登录页不能只在大屏成立

### 7.2 可访问性

要求：

- 颜色对比度必须可读
- 焦点态必须可见
- 关键按钮必须有明确 hover / focus 反馈

## 8. 新模块开发建议

当新增页面或组件时，建议按以下顺序判断：

1. 它属于主工作台、探索页、入口页还是沉浸层
2. 它应该继承哪一类已存在页面的视觉语言
3. 它是否破坏了当前的色彩、字体、容器和层级规则
4. 它是否会让工作台与沉浸层的边界重新混乱

如果一个新模块无法直接归类，优先参考：

- 聊天页：主舞台
- 技能页：探索页
- 登录页：品牌入口
- 执行状态工作台：沉浸层

### 8.1 主题接入规则

新增模块必须遵循以下主题接入顺序：

1. 先复用 `src/theme/palette.ts` 中已有语义，而不是直接新增裸色值
2. 先使用 `src/styles/variables.less` 暴露的语义 token，而不是在模块 less 中直接写 `rgba(...)`
3. 如果组件依赖 Ant Design portal 或 token，优先通过 `createAntdTheme(mode)` 的语义映射补齐
4. 如果组件处于图表、Markdown、代码块等富内容层，必须同时验证 `dark` 与 `light` 的可读性
5. 沉浸式 `Star-Office-UI` iframe 内部内容不强制跟随主题重绘，只改外层承载 chrome

## 9. 实施建议

新增 UI 功能时推荐流程：

1. 先读取本文档与 `src/styles/variables.less`
2. 优先复用现有 token，而不是先写死颜色值
3. 优先继承现有组件层级，而不是另起一套容器语言
   特别注意先判断这是“框架级结构”还是“内容级舞台”，不要把整页再次包成新的大壳体
4. 对新增页面补最少必要测试，防止视觉改动连带破坏交互

- 全局视觉 token：
  - `src/styles/variables.less`

  - `src/theme/palette.ts`
- 运行时主题状态：
  - `src/stores/useThemeStore.ts`
- Ant Design 双主题映射：
  - `src/theme/antdTheme.ts`
- 全局基础样式：
  - `src/styles/global.less`
- 主聊天工作台：
  - `src/pages/Chat/index.tsx`
  - `src/pages/Chat/Chat.module.less`
- 技能页：
  - `src/pages/Skills/index.tsx`
  - `src/pages/Skills/Skills.module.less`
- 登录页：
  - `src/pages/Login/index.tsx`
  - `src/pages/Login/index.module.less`
- 执行状态工作台：
  - `src/components/Visualize/VisualizeWorkbenchFrame.tsx`
