# agent-chat UI 视觉系统指南

## 1. 文档目的

本文档用于沉淀 `agent-chat` 当前阶段已经确认并落地的 UI 视觉系统，作为后续新增页面、组件与功能模块时的统一参考。

目标不是限制实现自由度，而是避免以下问题反复出现：

- 新页面重新回到“默认后台风”
- 新组件沿用另一套颜色、圆角、阴影和信息层级
- 主聊天工作台、技能页、登录页、执行状态工作台之间再次出现风格断层
- 只做局部好看，整体气质重新发散

## 2. 视觉主轴

当前项目采用的唯一主视觉方向为：

> `Paper Ops / 智能工作台`

核心判断：

- 主应用不是科技霓虹风，也不是通用蓝白后台风
- 主界面强调“纸面工作台、档案感、批注感、编辑式推进”
- 沉浸式 `Star-Office-UI` 像素办公室属于产品的“沉浸层”，不是主应用的直接复制对象

因此需要始终保持：

- 主界面：克制、专业、可长时间使用
- 沉浸层：保留更强的风格表达和戏剧性

## 3. 页面层级原则

### 3.1 主工作台优先

聊天页是产品主舞台，设计优先级最高：

- 会话标题、上下文说明、消息画布、输入区要形成一个完整工作台
- 不要把聊天页做成普通“左侧栏 + 中间列表 + 下方输入框”的通用套壳

### 3.2 子页面必须继承主叙事

技能页、登录页、执行状态工作台都必须属于同一产品体系：

- 技能页是“能力扩展页”，不是普通卡片列表
- 登录页是“品牌入口页”，不是只放一个登录按钮
- 工作台是“沉浸层”，不是把主应用再复制一遍

### 3.3 工作台与沉浸层分工

执行状态相关体验遵循以下边界：

- 聊天页中的执行状态入口应轻量明确
- 沉浸式工作台坚持 `iframe-first`
- 工作台外层控件极简，不得反向挤占真实像素办公室画面

## 4. 设计 Token 约束

### 4.1 色彩

主色：

- 主品牌色使用偏稳重的靛蓝系
- 局部强调使用铜棕色系

禁止：

- 回到标准高饱和蓝白后台配色
- 大面积使用没有上下文的紫色渐变
- 为了“科技感”堆叠高亮霓虹色

### 4.2 字体

当前约束：

- 标题优先使用更具识别度的 display serif 风格
- 正文保持高可读 sans-serif
- 代码与技术内容使用等宽字体

新增模块时：

- 不要把所有标题都退回系统 sans-serif 粗体
- 不要为局部页面单独引入另一套风格完全不同的字体语言

### 4.3 容器语言

统一采用：

- 暖白/纸白表面层
- 轻边框
- 克制阴影
- 圆角偏柔和但不夸张

禁止：

- 大面积纯白卡片 + 灰边的默认 SaaS 卡片感
- 所有区域都使用相同容器导致信息层级消失

## 5. 组件设计约束

### 5.1 布局组件

适用于：

- `Layout`
- 页面头部
- 区域面板

要求：

- 通过留白、信息密度和轻质感背景建立层级
- 主次区域必须明显，不要平均分配视觉注意力

### 5.2 聊天消息组件

要求：

- 用户消息与助手消息必须有明确角色差异
- 图表、卡片、文件结果必须拥有独立结果容器语义
- 时间戳、消息操作、状态反馈应贴边收纳，不要成为漂浮附件

禁止：

- 把所有消息都做成一模一样的白色方卡
- 图表和卡片退化成普通文本块

### 5.3 输入组件

要求：

- 输入区要像“批注条”或“工作台操作条”
- 文件上传预览与文本输入必须属于同一表面层
- 焦点态要清晰可感知

### 5.4 技能组件

要求：

- 技能卡要体现“能力模块”而不是普通商品卡
- 搜索、筛选、详情抽屉要和主视觉系统一致

### 5.5 工作台控件

要求：

- 沉浸式工作台悬浮控件必须极简
- 允许有返回、收起、刷新
- 不允许重新铺大面积说明文字或大头部占位

## 6. 动效与反馈

允许：

- 页面进入时的轻量过渡
- 卡片 hover 与消息出现的细微位移
- 输入焦点、按钮 hover、状态切换的清晰反馈

禁止：

- 长时间持续动画
- 与业务无关的炫技式动效
- 大范围闪烁、强缩放、无意义浮动

## 7. 响应式与可访问性

### 7.1 响应式

优先级：

1. 桌面宽屏
2. 中等宽度桌面
3. 再向下收敛移动端

要求：

- 聊天主舞台不能在中等宽度下崩塌
- 技能页、欢迎页、登录页不能只在大屏成立

### 7.2 可访问性

要求：

- 颜色对比度必须可读
- 焦点态必须可见
- 关键按钮必须有明确 hover / focus 反馈

## 8. 新模块开发建议

当新增页面或组件时，建议按以下顺序判断：

1. 它属于主工作台、探索页、入口页还是沉浸层
2. 它应该继承哪一类已存在页面的视觉语言
3. 它是否破坏了当前的色彩、字体、容器和层级规则
4. 它是否会让工作台与沉浸层的边界重新混乱

如果一个新模块无法直接归类，优先参考：

- 聊天页：主舞台
- 技能页：探索页
- 登录页：品牌入口
- 执行状态工作台：沉浸层

## 9. 实施建议

新增 UI 功能时推荐流程：

1. 先读取本文档与 `src/styles/variables.less`
2. 优先复用现有 token，而不是先写死颜色值
3. 优先继承现有组件层级，而不是另起一套容器语言
4. 对新增页面补最少必要测试，防止视觉改动连带破坏交互

  - `src/pages/Chat/index.tsx`
  - `src/pages/Chat/Chat.module.less`
- 技能页：
  - `src/pages/Skills/index.tsx`
  - `src/pages/Skills/Skills.module.less`
- 登录页：
  - `src/pages/Login/index.tsx`
  - `src/pages/Login/index.module.less`
- 执行状态工作台：
  - `src/components/Visualize/VisualizeWorkbenchFrame.tsx`
  - `src/components/Visualize/VisualizeWorkspaceView.module.less`
