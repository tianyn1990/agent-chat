# 阶段六：真实 Star-Office-UI 侧车接入方案

> 目标：在不牺牲真实像素办公室效果的前提下，把 Star-Office-UI 作为独立服务接入 `agent-chat`，并保持多会话语义、同域嵌入体验与后续可维护性。

## 1. 为什么需要这份文档

阶段六当前已经做了两件正确的事：

- 前端已经具备“查看执行状态”的入口、右侧面板和独立页面承接能力
- 本地已经具备 `npm run dev` 可用的 mock 方案，用来验证 iframe 与多会话绑定

但用户提出了一个更高要求，而且这个要求是合理的：

- 不是只要一个能动的 mock 页面
- 而是希望看到 **真实 Star-Office-UI** 的像素办公室效果

这一点非常关键，因为：

- Star-Office-UI 的价值很大一部分就在于其像素风办公室表现
- 如果本地看到的只是抽象状态块，那么对产品体验的判断就会失真
- 前端即使验证了协议，也没有真正验证最终视觉承接效果

因此，现有本地 mock 不能替代真实接入方案，它只能作为兜底和测试夹具。

## 2. 结论先行

### 2.1 推荐结论

推荐采用：

> **真实 Star-Office-UI 侧车服务 + 会话适配层 + 同域 iframe**

这是当前唯一同时满足以下目标的路线：

- 保留真实像素办公室效果
- 保持用户在 `agent-chat` 内沉浸式查看
- 支持后续会话级状态语义
- 保持与上游项目的升级边界

### 2.2 不推荐的路线

不推荐：

- 继续扩大本地假 mock 页面，试图“长得越来越像”
- 把 Star-Office-UI 全量重写成 React 组件
- 直接把上游源码和素材硬拷贝进主应用构建链路

## 3. 上游项目的真实形态

根据当前调研，Star-Office-UI 不是组件库，而是完整应用：

- 后端为 `Flask`
- 前端为原生静态页面
- 动画与场景逻辑依赖 `Phaser`
- 通过 `/status`、`/agents` 等接口读取运行态

这意味着：

- 它更像一个“可嵌入的网站”
- 而不是一个可以直接 `import` 的 UI 组件

所以最自然的接入方式就是：

- **把它当作独立服务运行**
- **再通过 iframe 嵌入当前页面**

## 4. 当前本地 mock 的角色重新定位

现有本地 mock 方案仍然有价值，但它的角色必须说清楚：

- 它负责验证：
  - iframe 承接
  - 前端会话绑定
  - 执行状态入口
  - 回归测试和无 sidecar 场景兜底
- 它不负责：
  - 提供真实像素办公室效果
  - 作为最终用户长期看到的视觉形态

所以未来阶段六文档里要同时保留两条路线：

1. **local mock 路线**
   - 用于开发期兜底和自动化验证
2. **real sidecar 路线**
   - 用于真实视觉体验、本地高保真联调和最终落地

## 5. 推荐架构

### 5.1 总体拓扑

```text
浏览器
  └─ agent-chat
      ├─ 会话与消息界面
      ├─ 右侧执行状态面板
      └─ iframe -> Star-Office-UI

可视化适配层
  ├─ 维护 chat session 与 OpenClaw session 绑定
  ├─ 维护会话级最新执行状态
  ├─ 输出 Star-Office 兼容接口
  └─ 为 iframe 提供按会话查看的 facade

Star-Office-UI sidecar
  ├─ 真实像素办公室页面
  ├─ 真实 Phaser 动画逻辑
  └─ 轮询状态接口

OpenClaw / Gateway
  └─ 提供真实执行事件与 session 上下文
```

### 5.2 核心职责分工

`agent-chat`：

- 负责“从哪个会话打开执行状态”
- 负责渲染 iframe 容器
- 负责在配置缺失时明确降级

服务端适配层：

- 负责回答“这个会话当前是什么状态”
- 负责把 OpenClaw 事件翻译为 Star-Office 状态机
- 负责屏蔽 Star-Office 原生无 session 的问题

Star-Office-UI：

- 负责真实视觉效果
- 不负责理解聊天业务语义

## 6. 为什么 iframe 方案现在是可行的

此前担心 iframe，主要是担心三件事：

1. 跨域
2. `postMessage` 不可控
3. 无法表达会话语义

现在看，正确解法是：

- **同域部署或受控子域部署** 解决访问问题
- **服务端适配层** 解决会话语义问题
- **不再要求 postMessage 承担业务协议职责**

也就是说，iframe 方案本身没有问题。

问题从来都不是 iframe，而是：

- 之前把“会话语义”错误地放在了前端通信层

现在把它改回服务端适配层后，iframe 就变成了合理的展示容器。

## 7. 多会话下到底展示哪个会话

这是整个方案最容易说不清的地方。

### 7.1 正确答案

当前产品语义应该定义为：

- 用户是从某个聊天会话里点击“查看执行状态”
- 所以这个面板必须绑定该会话的 `sessionId`
- iframe 中展示的，应该是 **这个会话对应的执行状态**

### 7.2 为什么 Star-Office-UI 原生做不到

因为它原生只有：

- 主 Agent 状态
- 访客 Agent 状态

它没有：

- 会话 ID
- 会话绑定
- “当前焦点会话”概念

### 7.3 所以必须有 session facade

要做到“点击哪个会话，就看哪个会话”，就必须存在一层 **session facade**：

- 它知道当前 iframe 对应的 `sessionId`
- 它知道这个 `sessionId` 映射到哪个 OpenClaw session
- 它给 Star-Office 提供该会话专属的状态接口

没有这层 facade，就不能声称支持会话级执行状态。

## 8. 本地开发如何看到真实像素办公室

### 8.1 推荐本地运行形态

本地推荐采用三段式：

1. `agent-chat` 开发服务器
2. 真实 Star-Office-UI sidecar
3. 本地 visualize adapter

推荐端口关系：

- `agent-chat`: `http://localhost:3000`
- Star-Office sidecar: `http://localhost:19000`

然后由开发代理把：

- `/star-office/` 转发到真实 Star-Office sidecar

前端则统一使用：

```env
VITE_STAR_OFFICE_URL=/star-office/
```

这样本地与生产都保持“同域 iframe”形态。

### 8.2 为什么不能只起一个真实 Star-Office 服务

因为只起服务只能解决“画面是真的”，不能解决“当前看哪个会话”。

所以本地仍然需要：

- mock gateway 或本地事件源
- visualize adapter
- session facade

### 8.3 本地需要保留 mock-fallback

即使后续接入真实 sidecar，也建议保留当前本地 mock：

- 当 sidecar 没有启动时，前端仍然可以继续开发
- 自动化测试不必强依赖外部 Python 服务
- 回归验证时能快速判断问题在“前端承接”还是“真实 sidecar / 适配层”

## 9. 生产部署怎么做

### 9.1 推荐总体方向

生产环境推荐两种可接受部署形态：

1. **子域名**
2. **同域子路径**

### 9.2 子域名方案

示例：

- `https://chat.example.com`
- `https://office.example.com`

优点：

- 对上游前端侵入较小
- 不强依赖 base path 改造
- 更适合作为独立 sidecar 运维

缺点：

- 域名、嵌入策略和安全头处理更复杂

### 9.3 同域子路径方案

示例：

- `https://example.com/`
- `https://example.com/star-office/`

优点：

- 用户体验更统一
- iframe 地址更自然

缺点：

- 上游前端如果默认请求根路径，可能需要补 patch 或代理重写

### 9.4 当前建议

如果实现阶段要尽量少改上游：

- 优先子域名

如果产品明确要求“相同域名内嵌，路径也统一”：

- 采用同域子路径，但要接受维护少量 patch 或 facade 代理

## 10. 仓库管理与交付边界

### 10.1 推荐方式

推荐优先级：

1. 公司自有 fork
2. `git submodule`
3. `git subtree`

### 10.2 不建议直接复制源码

原因：

- 无法优雅升级
- 不利于保留许可证边界
- 容易让主仓库承担无谓体积与维护成本

### 10.3 为什么推荐自有 fork

因为我们大概率需要维护少量接入差异，例如：

- base path 适配
- session facade 注入
- 本地开发运行脚本

这些改动如果全靠临时 patch，后续会越来越难维护。

## 11. 许可证风险必须单列

这是这份文档必须强调的重点。

根据当前调研，Star-Office-UI 的代码与像素素材并不是完全相同的许可边界。

工程上必须按以下原则处理：

- 在实现前完成许可证核查
- 在可能存在商业交付时，不能默认直接使用上游像素素材
- 如果存在风险，必须提前设计：
  - 获取授权
  - 或替换为可商用素材

这也是为什么不建议把真实素材直接打进主应用 bundle：

- 一旦混入主包，后续替换和合规边界都会变得更糟

## 12. 推荐落地顺序

### 第一步：冻结来源

- 决定自有 fork / submodule / subtree 方式
- 固定上游 commit

### 第二步：确认许可证

- 明确当前交付是否涉及商业使用
- 明确是否需要授权或替换素材

### 第三步：打通本地真实 sidecar

- 能启动真实 Star-Office 服务
- 能通过 iframe 看到真实像素办公室

### 第四步：补 session facade

- 让不同会话真正能看到自己的执行状态

### 第五步：接入真实适配层

- 从 OpenClaw 或业务后端同步真实状态

### 第六步：补生产部署

- 子域名或子路径最终定稿
- Docker / Nginx / 运维说明落地
