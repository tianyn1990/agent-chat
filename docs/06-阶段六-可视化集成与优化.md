# 阶段六：可视化集成与多会话校正

> **当前状态**：🚧 进行中。已完成摘要侧栏、沉浸式工作台、同会话 iframe 保活、多会话发送态修正；服务端会话适配与生产部署待接入。

## 1. 阶段目标

- 在对话界面内嵌 Star-Office-UI，提供沉浸式执行状态面板
- 明确 `agent-chat sessionId` 与 OpenClaw session 的关系，避免“伪多会话”
- 将错误的 `iframe + postMessage` 设想修订为真实可落地的同域部署方案
- 修复前端中阻碍多会话的基础问题，例如全局发送态
- 继续推进性能优化与文档完善

## 2. 关键判断

### 2.1 OpenClaw 支持多 session，但前提是后端真有映射

OpenClaw 官方文档明确存在 session 模型，并保证：

- 同一 session 内任务串行，避免上下文竞争
- 不同 session 可以并行执行
- gateway 才是 session 的唯一事实来源

因此，前端的多会话设计方向没有错，但必须补齐后端契约：

- `agent-chat` 中的 `sessionId`
- OpenClaw gateway 内部的 `sessionKey / sessionId`
- 可视化模块要看的“当前会话执行状态”

如果后端没有把这三者映射起来，前端就只是“界面上有多个会话”，而不是“系统里真的有多个隔离上下文”。

### 2.2 Star-Office-UI 不是 workflow DAG 组件

Star-Office-UI 的真实能力是：

- 展示主 Agent 的状态 `/status`
- 展示多个访客 Agent 的状态 `/agents`
- 用像素办公室表现 `idle / researching / writing / executing / syncing / error`

它原生不具备：

- session 概念
- 某个会话独享的执行流
- 基于 `postMessage` 接收 `sessionId` 后自行查询工作流
- 节点图 / DAG / trace 还原

所以阶段六必须把文案和语义改正：

- 面板本质上是“执行状态视图”
- 会话级语义依赖服务端适配层，而不是 Star-Office-UI 前端自身

## 3. 新的集成方案

### 3.1 架构

```text
agent-chat 前端
    └─ iframe 嵌入 Star-Office-UI
          └─ 同域部署（推荐）或独立部署

OpenClaw Gateway / 自定义后端适配层
    └─ 将会话状态翻译为 Star-Office-UI 可识别的状态推送
```

### 3.2 为什么不再使用 postMessage

原文档中的 `postMessage` 协议已废弃，原因：

- Star-Office-UI 并不是为嵌入式组件设计的
- 它不依赖浏览器事件驱动状态，而依赖后端接口 `/status` `/agents`
- 即便 iframe 同域，真正决定“看哪个会话”的也不是前端通信，而是服务端适配逻辑

### 3.3 推荐部署方式

推荐部署到与 `agent-chat` 相同域名下，例如：

- `https://example.com/chat/...`
- `https://example.com/star-office/`

这样 iframe 不会碰到常见跨域阻碍，也便于统一登录、运维和页面体验。

但是有一个实际限制必须提前说明：

- Star-Office-UI 当前前端代码默认请求根路径 `/status`、`/agents`
- 如果要部署在 `/star-office/` 这样的路径前缀下，需要额外处理 base path 问题
- 如果不愿修改上游项目，可以改用子域名，例如 `https://office.example.com`

## 4. 会话语义

### 4.1 我们采用的语义

当前阶段的前端交互定义为：

- 用户从某条 AI 回复点击“查看执行状态”
- 面板记录当前 `sessionId`
- 面板标题和状态文案都以该会话为上下文
- iframe 中真正展示什么，由服务端适配层决定

这比“无论点哪个会话都看同一个全局办公室”更符合用户直觉。

### 4.2 为什么不能直接说“查看工作流”

Star-Office-UI 当前更像状态看板，而不是工作流图。

因此本阶段的 UI 文案调整为：

- `查看执行状态`

未来如果后端能提供 trace / node graph，再考虑升级成：

- `查看工作流`

## 5. 已完成实现

### 5.1 前端骨架与交互流重构

- 对话页右侧面板已收敛为“执行状态摘要侧栏”，不再在窄侧栏内强塞真实像素办公室
- 助手消息新增 `查看执行状态` 入口，可直接绑定当前会话上下文
- 新增 `/visualize/:sessionId` 沉浸式工作台路由，真实 Star-Office-UI 在该工作台中完整展示
- `VisualizeWorkbenchHost` 作为全局常驻宿主保活真实 iframe，关闭时只隐藏，不销毁
- 再次打开同一会话的工作台时，会直接复用已初始化的页面，避免重复冷启动
- `VITE_STAR_OFFICE_URL` 改为显式配置；未配置时给出部署提示，而不是错误地假定本地 3001

### 5.2 多会话基础修正

已修复一个关键问题：

- 过去前端使用全局 `isSending`
- 任一会话在发送/流式回复时，会锁住所有会话

现已改为：

- 按 `sessionId` 维护发送态
- 当前会话只受自身发送态影响

这一步是支持多会话的必要前提。

### 5.3 会话级可视化状态缓存

前端已增加一层本地运行态缓存，用于标记某个会话当前大致处于：

- `researching`
- `writing`
- `idle`
- `error`

这层状态当前仅用于面板文案与前端反馈，不替代后端真实状态。

### 5.4 本地开发 Mock 已落地

本轮已补齐 `npm run dev` 可直接验证的本地 mock 链路：

- 新增同域 iframe 承接页面 `/__mock/star-office/app`
- 新增 Vite dev server 内存态 adapter 接口
- 新增前端会话运行态到本地 adapter 的桥接逻辑
- 本地 mock 严格按 `sessionId` 隔离状态，避免多会话串线

推荐本地配置：

```env
VITE_MOCK_ENABLED=true
VITE_STAR_OFFICE_URL=
VITE_STAR_OFFICE_MOCK_ENABLED=true
VITE_STAR_OFFICE_MOCK_BASE=/__mock/star-office
```

这样在本地点击“查看执行状态”时，会优先加载本地 mock 页面，而不是空白 iframe。

### 5.5 本地真实 Star-Office 联调已落地

为解决“本地只能看到假 mock，无法看到真实像素办公室效果”的问题，现已补齐开发期真实联调能力：

- 新增会话级真实 facade 入口：`/star-office/session/:sessionId/`
- Vite dev server 可直接承接真实上游 `Star-Office-UI` 前端资源
- 在会话级入口内对上游 `index.html` 做最小注入，重写 `/status`、`/agents`、`/static/*` 等根路径请求
- 继续复用本地 adapter store 作为会话级状态来源，避免为了本地联调再起一套后端
- 当上游目录缺失或资源不可用时，iframe 内会显示明确诊断页，而不是空白页

推荐本地配置：

```env
VITE_STAR_OFFICE_URL=
VITE_STAR_OFFICE_REAL_DEV_ENABLED=true
VITE_STAR_OFFICE_REAL_DEV_BASE=/star-office
VITE_STAR_OFFICE_UPSTREAM_DIR=/tmp/Star-Office-UI
VITE_STAR_OFFICE_MOCK_ENABLED=true
```

当前优先级：

1. `VITE_STAR_OFFICE_URL`
2. `VITE_STAR_OFFICE_REAL_DEV_ENABLED=true`
3. `VITE_STAR_OFFICE_MOCK_ENABLED=true`

### 5.6 已完成的性能优化

本轮已完成的性能项：

- `TableChart` 与 `G2Chart` 继续保持按需懒加载
- `vite` 已补充手动 chunk 拆分，重型图表与 Markdown 依赖不再进入首屏主包
- 聊天中的图片预览与图表图片已补充原生 `loading="lazy"`
- 对话核心链路中的 Markdown 渲染已恢复为同步装载，避免高负载或测试并发时长期停留在回退态，保证消息展示稳定性

当前构建结果（2026-03-24）：

- `vendor-markdown`: 约 `232 KB`，较之前约 `785 KB` 明显下降
- `vendor-g2`: 约 `1304 KB`
- `vendor-antd`: 约 `1216 KB`

说明：

- `vendor-antd` 仍偏大，但属于当前 UI 框架主包成本，不会阻塞阶段六交付
- 这部分如果继续深挖，应放在后续阶段以“组件级替换 / 按路由继续切分”为主题单独处理

### 6.4 沉浸式工作台流程变更

为解决“真实像素办公室在右侧面板里展示不完整、交互受限、反复打开要重复初始化”的问题，当前已采用新的工作流：

- 聊天页右侧只保留摘要侧栏，用于说明会话状态和进入工作台
- 真实 Star-Office-UI 进入独立沉浸式工作台展示，保证更完整的视口和交互空间
- 工作台关闭时仅隐藏，不销毁真实 iframe
- 如果用户再次打开同一会话的工作台，直接展开已保活页面，避免重新初始化

### 6.5 一键直达与 iframe-first 收敛

在上一轮“摘要侧栏 + 沉浸式工作台 + 保活隐藏”的基础上，本轮又进一步收敛了交互：

- `查看执行状态` 已从消息级动作区迁移到聊天页会话头部，明确成为当前会话的固定能力入口
- 会话头部入口采用“像素工位徽章”样式，只借用像素办公室的识别记忆点，不把主界面做成游戏化按钮
- 聊天页不再默认展开大面积执行状态侧栏，只保留轻量恢复提示
- 工作台关闭后会自动回到聊天页，同时留下一个小型恢复入口
- 沉浸式工作台已改成 `iframe-first` 布局，真实 `Star-Office-UI` 占满主体视口
- 应用内返回、收起、刷新控件已收敛为悬浮在 `iframe` 上层的工具条，不要求用户依赖浏览器原生返回
- 当前活跃会话进入聊天页后会静默预热工作台，首次点击时由外层加载封面承接冷启动
- 工作台缓存已从“同会话单实例保活”升级为 `LRU=2`，覆盖最近两个会话的快速往返恢复
